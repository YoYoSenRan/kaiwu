# 设计原则

本文件约束代码层面的设计质量：模块怎么拆、依赖怎么管、边界怎么画。**所有新增/修改代码都必须过以下原则。**

## 1. 高内聚，低耦合

- **高内聚**：一个文件里的所有代码必须服务于**同一抽象层级、同一业务概念**。出现网络协议 + UI 渲染 + 文件 IO 混写 = 必须拆。
- **低耦合**：模块之间通过**窄接口**通信，不直接访问彼此内部状态。禁止 `a.b.c.internal.ws.readyState` 这种深入调用。

## 2. 单一职责原则（SRP）

- 一个函数、一个文件、一个模块**只对一个变化原因负责**。
- 判断标准：如果改需求 A 时要顺便改这个地方，改需求 B 时又要改同一个地方——说明职责没拆清。

## 3. 关注点分离

- **传输层**（怎么连）和**业务层**（发什么）分离。
- **主进程**代码和 **preload** 代码分离——preload 禁止 import `ipcMain` / `BrowserWindow` / `service.ts`。
- **状态管理**、**副作用**、**纯渲染**在 renderer 侧也要分层。

## 4. 依赖倒置

- **上层依赖接口，不依赖具体实现**。
- 例子：`service.ts` 应该依赖 `GatewayClient` 类型，而不是直接 `new WebSocket()`。
- feature 之间禁止互相 import；共享能力下沉到 `core/`。

## 5. 开闭原则（OCP）

- **对扩展开放，对修改关闭**。
- 加一个 RPC 方法，应该在 `methods.ts` 里加一行映射，而不是去改 `client.ts` 的协议解析逻辑。
- 加一个新 feature，应该是新增一组文件 + 在 `main.ts`/`preload.ts` 注册，而不是改十个现有文件。

## 6. 封装

- **内部状态藏起来，只暴露最小操作面**。
- 闭包变量、私有函数、模块级单例都不应该被外部直接读写。
- 禁止把内部数据结构（如 `Map`、`Set`、原始 WS 实例）直接暴露给调用方。

## 7. 正交性

- **两个正交的功能应该互不影响**。
- 改 UI 主题不应该触发网络重连；改日志级别不应该影响窗口状态。
- 如果发现改 A 导致 B 异常，说明正交性被破坏，要拆。

## 8. 迪米特法则（最少知识）

- **不要跟陌生人说话**。只调用直接依赖对象的方法，不跨层访问。
- ❌ `connection.client.ws.readyState`
- ✅ `connection.isConnected()`

## 9. 禁止重复机制互掐

- 如果下层已有重连/轮询/缓存机制，上层就不要再套一层同类型的机制。
- 例子：`client.ts` 内部有指数退避重连，`service.ts` 就不应该再用定时器强行断线重建。
- 两层同职责的保活/重试同时存在 = 必须合并或取消一层。

## 10. 快速失败，不要掩盖错误

- 错误在**最早发生的位置**暴露，不要加 fallback 让问题延后炸。
- 系统边界（用户输入、外部 API、IPC）做校验；内部模块之间**信任契约**，不做防御性 if。

---

**自检口诀**：新增一个功能前问自己——

1. 这件事该放在这个文件里吗？（内聚）
2. 改它需要动多少其他文件？（耦合）
3. 如果以后加类似能力，是写新代码还是改旧代码？（开闭）
4. 外部需要知道这个模块的内部细节吗？（封装）

# 整体架构与设计原则

## 1. Kaiwu 是什么

Kaiwu 是一条**内容生产流水线**——通过某种机制自动产生内容选题，经过多 Agent 协作审议和实施，最终自动发布。

三省六部是其中一种可替换的**治理叙事**，为整个流水线赋予故事感。底层的流水线逻辑与主题无关。

## 2. 核心数据流

```
灵感来源              审议 & 实施                          发布
(proposals)          (productions)                       (publications)
┌──────────┐  通过   ┌────────────────────────────────┐   ┌───────────┐
│ 自动发现  │───────▶│ planning → review → execute    │──▶│ 自动部署   │
│ 手动输入  │        │     ↑ 驳回  │                  │   │ 发布记录   │
│ 外部信号  │  驳回   │     └──────┘                  │   │ 最终 URL  │
│          │◀───────│                                │   └───────────┘
└──────────┘        │ 子任务拆分 (production_tasks)    │
                    │ 全程审计 (production_stages)     │
                    │ 事件追踪 (production_events)     │
                    └────────────────────────────────┘
```

## 3. 设计原则

### 3.1 逻辑与展示分离

数据库中不存储"中书省""门下省"这类主题专属词汇。所有流水线逻辑使用通用的 `stage_type`（如 `planning`、`review`、`execute`）。主题相关的名称、emoji、色值存在 `themes` 和 `pipelines` 表中，渲染时查表替换。

**好处：** 换主题 = 换一套 theme 数据，零代码改动。

### 3.2 proposals 与 productions 分离

选题（proposals）和正式立项（productions）是两张独立的表。

**为什么不合并：**

- proposals 可能大量产生、大量被驳回，是"候选池"
- productions 是经过审议的正式作品，有完整的生命周期管理
- 分离后 productions 表保持干净，查询性能更好
- 未来 proposals 的来源机制可能频繁调整，不影响核心流水线

### 3.3 产出物不入库

Agent 执行后的产出（报告、代码、数据）写入指定目录，数据库只存路径引用（`output_dir` 字段）。最终通过 Agent 自动部署发布。

**为什么：**

- 产出物体积不可控（可能是大文件）
- 产出物需要直接被部署流程读取
- 数据库不适合存储二进制/大文本

### 3.4 审计优先

每一次状态变更都记录到 `production_stages`，每一个系统事件都写入 `production_events`。数据可以丢、可以重建，但审计链不能断。

### 3.5 最小表数量

不过度拆表。`agent_permissions` 和 `agent_skills` 不单独建表——Kaiwu 的流水线是固定顺序，权限规则硬编码在业务逻辑中；skills 作为 agents 表的 JSONB 字段存储。量大了再拆。

## 4. 表分组总览

| 分组         | 表数量 | 说明             |
| ------------ | ------ | ---------------- |
| 身份         | 2      | 用户登录与会话   |
| 主题与流水线 | 2      | 可替换的叙事系统 |
| Agent        | 1      | 角色定义         |
| 内容流水线   | 4      | 从选题到实施     |
| 发布         | 1      | 部署与发布追踪   |
| 可观测       | 1      | 事件流           |
| 系统         | 1      | 配置             |
| **合计**     | **12** |                  |

## 5. 命名规范

| 规则                               | 示例                               |
| ---------------------------------- | ---------------------------------- |
| 表名：复数、snake_case             | `productions`、`production_stages` |
| 主键：`id`（serial 或 uuid）       | `id: serial("id").primaryKey()`    |
| 外键：`{关联表单数}_id`            | `production_id`、`agent_id`        |
| 时间戳：`created_at`、`updated_at` | 所有表都有                         |
| JSONB 字段：用于灵活扩展           | `meta`、`config`、`checkpoints`    |
| 布尔字段：`is_` 前缀               | `is_enabled`、`is_archived`        |

# 功勋榜·轶事·宿敌谱

## 设计原则

- **零新增数据表**：全部从现有数据（events、agent_stats、phases、debates）聚合计算
- **零新增编排逻辑**：不给编排层加负担，用查询时实时聚合或复盘时顺带计算
- **对 Agent 有实际影响**：不只是展示装饰，通过 MEMORY.md 和 SOUL.md 动态段落影响行为
- **符合世界观**：命名、描述、呈现方式全部用开物局的语言

## 功勋榜

开物局里谁的功劳最大、谁的手艺最精、谁的嘴最利——功勋榜一目了然。

### 榜单设计

每个榜单只取**前三名**，不做完整排名——八个人的局，排名太细没意义，前三就够了。

| 榜名 | 衡量什么 | 计算方式 | 数据来源 |
|---|---|---|---|
| **造物功臣** | 参与鸣锣成功最多 | COUNT(events WHERE type='product_launched' AND agent_id=?) | events |
| **慧眼如炬** | 采风评分最准 | 游商采风评分与器物最终表现的相关度 | phases.output + retrospectives |
| **舌灿莲花** | 说客论点被掌秤采纳最多 | 裁决书中 optimistPoints 的引用次数 | phases.output（council） |
| **一剑封喉** | 诤臣单轮发言直接导致封存 | 封存的造物令中，掌秤裁决引用诤臣论点为主因的次数 | phases.output + events |
| **明镜高悬** | 掌秤裁决准确率最高 | 裁决结果与器物最终命运的吻合度 | phases.output + projects.status |
| **妙笔生花** | 画师蓝图导致返工最少 | 因蓝图问题导致匠人返工的次数（越少越好） | events（review_failed）+ tasks |
| **铁锤无瑕** | 匠人返工率最低 | 被试剑打回的任务比例 | tasks + events |
| **火眼金睛** | 试剑放行后零线上问题 | 放行的器物在 30 天内无回滚/无严重 bug | events + retrospectives |

### 计算时机

**不实时计算**。在以下时机更新：

1. **造物令完成时**（鸣锣或封存）：编排层顺带更新相关榜单数据
2. **复盘时**（7/30/90 天）：有了器物的实际表现数据后，更新准确率类榜单

榜单数据存在哪里？**不存**。每次展示网站请求时从现有数据实时聚合。八个人的数据量，一条 SQL 聚合查询毫秒级返回，不需要缓存。

### 对 Agent 的影响

编排层在更新 SOUL.md 动态段落时，加入功勋榜信息：

```markdown
## 当前状态（由编排层注入）

你目前是 Lv.3 老商期。
- 嗅觉 ★★★★☆
- 功勋榜「慧眼如炬」第 1 名——你的采风评分是局里最准的，继续保持
```

这不是虚荣心，是**正向强化**——Agent 知道自己哪方面做得好，会倾向于保持这个优势。

### 展示网站呈现

功勋榜放在**局中人总览页**（/agents）的顶部：

```
┌─────────────────────────────────────────────────────┐
│  功勋榜                                               │
│                                                     │
│  造物功臣    🔨 匠人(7) · 🎒 游商(7) · 🖌 画师(6)   │
│  慧眼如炬    🎒 游商 · 采风评分准确率 87%             │
│  舌灿莲花    🗣 说客 · 论点被采纳 23 次               │
│  一剑封喉    ⚔️ 诤臣 · 单轮封存 4 次                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 局中轶事

轶事是局中人的**成就徽章**——做到了某件不容易的事，就解锁一个轶事，永久记录在角色档案里。

### 设计约束

- 总共不超过 **20 个轶事**（太多就不稀缺了）
- 每个角色 2-3 个专属轶事 + 少量通用轶事
- 解锁条件必须**可从现有数据判断**，不需要额外埋点

### 轶事列表

#### 游商专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **千里眼** | 连续 3 次采风评分与最终结果偏差 < 10 分 | phases.output + retrospectives |
| **先知** | 在采风报告中提到的风险，后来在试剑阶段被验证 | agent_logs + phases.output |

#### 说客专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **翻盘王** | 在第 3 轮或之后反击成功，逆转局势 | debates（round ≥ 3 且最终通过） |
| **百发百中** | 连续 5 个支持的造物令最终鸣锣 | phases.output + projects.status |

#### 诤臣专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **铁嘴** | 连续 5 场过堂提出被掌秤采纳的关键质疑 | phases.output（council） |
| **心服口服** | 说客在过堂中主动承认"这次我没什么好反对的" | debates.content 关键词匹配 |

#### 掌秤专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **明镜** | 连续 10 次裁决与最终结果一致 | phases.output + projects.status |
| **有条件的智慧** | 附带条件的裁决中，条件被验证为必要的比例 > 80% | retrospectives |

#### 画师专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **一笔成图** | 蓝图零返工，匠人完全按图施工无偏差 | tasks（无 blocked/cancelled） |

#### 匠人专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **不眠匠** | 单个造物令中连续完成 10 个任务无返工 | tasks + events |
| **巧夺天工** | 试剑标注"超出预期"的次数 ≥ 3 | phases.output（inspector） |

#### 试剑专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **零漏** | 放行的器物在 30 天内零线上问题 | retrospectives |
| **铁面** | 连续 3 次打回匠人的产出（标准严格但最终器物质量极高） | events + retrospectives |

#### 鸣锣专属

| 轶事名 | 解锁条件 | 数据来源 |
|---|---|---|
| **一锤定音** | 连续 5 次部署零回滚 | events（无 smoke_test_failed） |

#### 通用轶事

| 轶事名 | 解锁条件 | 适用角色 |
|---|---|---|
| **开局元老** | 参与第一个造物令 | 所有 |
| **百炼成钢** | 参与 50 个造物令 | 所有 |

### 解锁机制

**在造物令完成时和复盘时检查**，和功勋榜同一时机，不额外增加计算。

```
造物令完成 / 复盘触发
  │
  ├── 编排层更新属性（已有逻辑）
  │
  ├── 顺带检查轶事解锁条件
  │   └── 条件满足 → 写入 events 表（type: achievement_unlocked）
  │                  写入 Agent 的 MEMORY.md 里程碑段落
  │
  └── SSE 推送到展示网站（弹出解锁通知）
```

轶事解锁记录存在 **events 表**（type: `achievement_unlocked`，detail 里存轶事 ID 和名称），不需要新表。

### 对 Agent 的影响

轶事写入 MEMORY.md 的里程碑段落：

```markdown
## 里程碑

- 造物令 #3：第一次被诤臣用漏掉的竞品数据反驳
- 造物令 #7：采风报告中的留存率数据成为过堂关键转折点
- 🏆 造物令 #10：解锁轶事「千里眼」——连续 3 次采风评分与最终结果偏差 < 10 分
```

Agent 看到自己解锁了"千里眼"，会强化"保持采风准确度"的行为倾向。

### 展示网站呈现

轶事在角色详情页（/agents/:id）展示为徽章墙：

```
┌─────────────────────────────────────────────────────┐
│  轶事                                                 │
│                                                     │
│  🏆 千里眼        🏆 先知        ○ 百炼成钢          │
│  造物令 #10 解锁   造物令 #15 解锁  （未解锁）        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

未解锁的轶事显示为灰色圆圈 + 解锁条件提示，已解锁的显示金色奖杯 + 解锁时间。

## 宿敌谱

说客和诤臣的对抗是开物局最大的看点。宿敌谱把这个对抗**量化、可视化、可回溯**。

### 数据结构

不需要新表。所有数据从 debates + phases.output（council 阶段）聚合：

```
宿敌谱数据 = {
  // 总战绩
  totalDebates: 23,
  shuikeWins: 12,        // 说客支持的造物令最终鸣锣
  zhengchenWins: 8,      // 诤臣质疑的造物令最终封存
  draws: 3,              // 有条件通过（双方都有道理）

  // 经典对决（局势反转最多的过堂）
  classicDebates: [
    { projectName: "极简记账", rounds: 3, turningPoint: "说客第 2 轮用留存率数据反击" },
    { projectName: "AI 客服", rounds: 6, turningPoint: "诤臣第 4 轮用合规风险一锤定音" }
  ],

  // 克制关系
  shuikeStrengths: ["留存率/弃用率数据", "历史成功案例类比"],
  zhengchenStrengths: ["合规风险", "获客成本分析"],

  // 最近 5 场
  recentResults: ["shuike", "zhengchen", "draw", "shuike", "shuike"]
}
```

### 计算时机

每次过堂结束时（掌秤裁决后），编排层顺带更新宿敌谱数据。数据不单独存表，查询时从 debates + phases.output 实时聚合。

### 对 Agent 的影响

宿敌谱数据写入说客和诤臣的 `memory/relationships.md`：

**说客的 relationships.md**：
```markdown
## 与诤臣（宿敌）

总战绩：12 胜 8 负 3 平
- 我的强项：用留存率/弃用率数据反击"市场饱和"论点（胜率 80%）
- 我的弱项：面对"合规风险"类质疑时缺乏有效反击（胜率 30%）
- 最近趋势：最近 5 场 3 胜 1 负 1 平，状态不错

下次过堂策略：
- 如果诤臣提"合规风险"，要提前准备合规方面的数据
- "留存率"这张牌已经用了很多次，诤臣可能有准备了
```

**诤臣的 relationships.md**：
```markdown
## 与说客（宿敌）

总战绩：8 胜 12 负 3 平
- 我的强项：用合规风险和获客成本分析质疑（胜率 90%）
- 我的弱项：面对留存率/弃用率数据时容易被反击（胜率 20%）
- 最近趋势：最近 5 场 1 胜 3 负 1 平，需要调整策略

下次过堂策略：
- 说客很可能又会用留存率数据，要提前查好反驳的角度
- 多从"获客成本"和"竞争壁垒"角度切入，这是我的优势区
```

这样下次过堂时，说客和诤臣会**真的根据历史交锋调整策略**——不是装的，是记忆驱动的。

## 实施策略

三个机制共享同一套实施路径，不分开做：

| 时机 | 做什么 |
|---|---|
| 造物令完成时 | 编排层已有的"属性计算"逻辑中，顺带计算功勋榜排名、检查轶事解锁、更新宿敌谱 |
| 复盘时（7/30/90 天） | 有了器物实际表现后，更新准确率类榜单和轶事 |
| 展示网站请求时 | 功勋榜和宿敌谱从现有数据实时聚合，不缓存 |

**不新增数据表**。轶事解锁记录存 events 表（type: achievement_unlocked）。功勋榜和宿敌谱纯查询聚合。

**不增加编排层复杂度**。所有计算都是在"造物令完成"这个已有的钩子里顺带做的，不是独立的流程。

# 造物流 — 阶段设计

## 全景

```
物帖投入 → [采风] → [过堂] → [绘图] → [锻造] → [试剑] → [鸣锣] → 器物问世
            游商    说客/诤臣  画师     匠人     试剑     鸣锣
                    /掌秤
```

**全自动，无人工干预**。局中人自行决策，掌秤是整条造物流的"最高裁决者"。

## 推进策略：心跳驱动

整条流水线由一个**全局心跳定时器**驱动。每次心跳触发时，编排层检查所有进行中的项目，推进一步。

```
心跳 tick
  │
  ├── 项目 A：当前在论证阶段 → 触发诤臣发表第 2 轮观点
  ├── 项目 B：当前在构建阶段 → 触发匠人完成下一个任务
  ├── 项目 C：当前在探测阶段 → 触发游商完成下一维度分析
  │
  └── SSE 推送所有变更到展示网站
```

### 心跳配置

```ts
interface HeartbeatConfig {
  /** 全局心跳间隔（毫秒），一个参数控制全局节奏 */
  interval: number

  /** 各阶段可覆盖全局间隔，精细调控节奏 */
  phaseOverrides?: {
    scout?: number       // 采风阶段间隔
    council?: number     // 论证阶段间隔（辩论轮次之间）
    architect?: number   // 规划阶段间隔
    builder?: number     // 构建阶段间隔（任务之间）
    inspector?: number   // 审查阶段间隔
    deployer?: number    // 发布阶段间隔
  }
}

// 示例：正式运行
{ interval: 30 * 60 * 1000 }  // 默认 30 分钟一次心跳

// 示例：调试模式
{ interval: 5 * 1000 }  // 5 秒一次，快速看完整个流程

// 示例：精细控制
{
  interval: 20 * 60 * 1000,       // 默认 20 分钟
  phaseOverrides: {
    council: 10 * 60 * 1000,      // 辩论轮次间 10 分钟（节奏更快，更有张力）
    builder: 30 * 60 * 1000,      // 构建任务间 30 分钟（模拟写代码的节奏）
  }
}
```

### 为什么用心跳

1. **节奏完全可控**——一个参数调快调慢，不用改代码
2. **展示体验好**——访客每次来都能看到"又推进了一步"，而不是"已经跑完了"
3. **运维友好**——想让流水线快跑追进度，拉短间隔；想慢慢展示，拉长间隔
4. **叙事线清晰**——一次只推进一个造物令，访客跟着一个故事看到底

### 每次心跳做什么

```
心跳触发
  │
  ▼
取当前 running 的造物令（同一时间只有一个）
  │
  ├── 检查当前阶段
  │     │
  │     ├── 阶段进行中 → 推进一步（发一轮辩论 / 完成一个构建任务 / ...）
  │     ├── 阶段刚完成 → 执行自动决策规则，决定进入下一阶段还是归入封存阁
  │     └── 阶段异常 → 重试或标记失败
  │
  ├── 如果当前造物令刚完成（鸣锣 or 封存）→ 从队列取下一个
  │
  ├── 写入事件日志
  │
  └── SSE 推送到展示网站
```

### 自动决策规则（不变，仍由 Agent 决策）

| 衔接点           | 规则                                     |
| ---------------- | ---------------------------------------- |
| 探测完成         | 评分 ≥ 60 进入论证，< 60 归入封存阁        |
| 论证完成         | 掌秤裁决"通过"进入规划，"否决"归入封存阁 |
| 规划完成         | 自动进入构建                             |
| 构建完成         | 自动进入审查                             |
| 审查不通过       | 回退构建修复，最多 3 轮                  |
| 审查通过         | 自动进入发布                             |
| 发布冒烟测试失败 | 回滚，回退审查                           |

## Phase 1 — 探测（Scout）

**目标**：快速评估一个关键词/创意是否值得深入。

### 输入

- 关键词或一句话创意描述

### Agent 工作内容

| 维度       | 具体动作                               |
| ---------- | -------------------------------------- |
| 市场调研   | 搜索竞品、分析市场规模、趋势走向       |
| 用户需求   | 目标用户画像、痛点分析、需求频次       |
| 技术可行性 | 技术栈评估、核心难点识别、预估开发周期 |
| 商业价值   | 变现路径、获客成本预估、护城河评估     |
| 综合评分   | 0-100 分，附带维度雷达图               |

### 输出

- 《采风报告》：结构化文档，包含上述所有维度
- 评分：综合分 + 各维度分
- 建议：🟢 推荐立项 / 🟡 需进一步论证 / 🔴 不建议

### 自动决策规则

- 综合评分 ≥ 60 → 自动进入论证阶段
- 综合评分 < 60 → 自动归入封存阁，生成封存辞
- 关键词过于模糊导致搜索无有效结果 → 标记为"无法评估"，归入封存阁

---

## Phase 2 — 论证（Council）

**目标**：多角度辩论，形成立项决策。掌秤是最终决策者。

### 参与 Agent

| 角色                    | 立场     | 关注点                             |
| ----------------------- | -------- | ---------------------------------- |
| 🟢 说客（Optimist）   | 支持做   | 市场机会、增长潜力、差异化空间     |
| 🔴 诤臣（Skeptic）    | 质疑做   | 风险、竞争壁垒、资源消耗、失败概率 |
| 🟡 掌秤（Pragmatist） | 最终仲裁 | 综合双方观点，做出最终裁决         |

### 辩论机制

1. **第一轮**：说客和诤臣各自基于采风报告发表初始观点
2. **第二轮**：针对对方观点进行反驳
3. **第三轮**：回应反驳，修正或坚持立场
4. **第四轮**（可选）：如果分歧仍然很大，进行最终陈述
5. **裁决**：掌秤综合所有论点，出具最终裁决

### 辩论规则

- 每轮发言必须引用具体数据或逻辑，禁止空洞断言
- 必须回应对方的核心论点，不能回避
- 每轮发言有字数上限，防止注水

### 输出

- 掌秤的最终裁决：✅ 通过（可附带条件）/ ❌ 否决
- 完整辩论记录（供展示网站回放）

### 自动决策规则

- 掌秤裁决"通过" → 自动进入规划阶段（附带的条件会传递给画师作为约束）
- 掌秤裁决"否决" → 自动归入封存阁，掌秤撰写封存辞

---

## Phase 3 — 规划（Architect）

**目标**：将立项决策转化为可执行的项目计划。

### Agent 工作内容

1. **产品定义**
   - 核心功能列表（MVP scope）
   - 用户故事（User Stories）
   - 功能优先级排序（MoSCoW）
   - 如果掌秤裁决附带条件，必须在规划中体现

2. **技术架构**
   - 技术栈选型与理由
   - 系统架构图
   - 数据模型草案
   - 第三方服务依赖

3. **任务拆解**
   - WBS（工作分解结构）
   - 任务依赖关系图
   - 各任务分配给哪个 Builder Agent

4. **风险预案**
   - 已识别风险 + 应对策略
   - 回退方案

### 输出

- 《项目规划书》：包含上述所有内容
- 任务列表：自动初始化到流水线系统中

### 自动流转

- 规划完成后自动进入构建阶段，任务列表分发给 Builder Agent

---

## Phase 4 — 构建（Builder）

**目标**：根据规划书，多 Agent 协作完成代码开发。

### Agent 分工

| Agent        | 职责                         |
| ------------ | ---------------------------- |
| 前端 Builder | 页面组件、样式、交互逻辑     |
| 后端 Builder | API、数据库、业务逻辑        |
| 设计 Builder | UI 决策、配色、布局规范      |
| 测试 Builder | 测试用例生成、自动化测试执行 |

### 工作模式

- 按任务列表逐个认领、完成、提交
- 匠人的子角色可并行工作（前后端同时开发，需无依赖关系）
- Agent 之间可以互相提问和协调
- 所有代码变更都有完整的 commit 记录和决策说明

### 过程追溯

- 每次 commit 关联到任务 ID
- Agent 的思考过程和决策理由记录在日志中
- 关键决策点（如技术方案变更）自动高亮

### 自动流转

- 所有任务完成后自动进入审查阶段

---

## Phase 5 — 审查（Inspector）

**目标**：确保产出质量达标。

### 审查清单

| 维度       | 内容                       |
| ---------- | -------------------------- |
| 代码质量   | Lint、类型检查、代码规范   |
| 安全性     | 依赖漏洞扫描、XSS/注入检查 |
| 性能       | Lighthouse 评分、加载时间  |
| 功能完整性 | 对照 MVP 功能列表逐项验证  |
| 用户体验   | 关键路径走查、边界场景覆盖 |

### 输出

- 《审查报告》：通过 / 不通过
- 问题清单：按严重程度排序

### 自动决策规则

- 🔴 严重问题 = 0 且 🟡 一般问题 ≤ 3 → 通过，自动进入发布
- 存在 🔴 严重问题 → 不通过，自动回退到构建阶段修复
- 回退修复后重新审查，最多 3 轮。3 轮仍不通过 → 标记为"构建失败"，归入封存阁

---

## Phase 6 — 发布（Deployer）

**目标**：将产品部署到生产环境，让用户可以访问。

### 部署流程

1. 构建产物打包
2. 部署到 Vercel
3. 域名配置
4. SSL 证书
5. 冒烟测试（自动访问关键页面，检查响应）
6. 监控接入（uptime、错误追踪）

### 输出

- 可访问的产品 URL
- 《发布报告》：部署信息、监控地址、回滚方案
- 展示网站自动更新：新增产品卡片和诞生故事

### 自动决策规则

- 冒烟测试通过 → 标记为"已上线"，造物志自动生成终章
- 冒烟测试失败 → 自动回滚，回退到审查阶段重新检查

## 全自动状态机

```
[投入] → [探测中] ──→ [论证中] ──→ [规划中] ──→ [构建中] ──→ [审查中] ──→ [发布中] → [已上线]
              │            │                          │            │
              ▼            ▼                          │            ▼
          [封存阁]       [封存阁]                    ┌────┘        [封存阁]
          (<60分)     (掌秤否决)               │          (3轮修复失败)
                                                │
                                           [修复中]
                                          (最多3轮)
```

每个状态变更都会产生一条事件记录，用于展示网站的时间线回放。

## 异常兜底

| 异常                     | 处理                                                       |
| ------------------------ | ---------------------------------------------------------- |
| Agent 调用超时           | 重试 1 次，仍超时则标记该阶段失败，归入封存阁                |
| Agent 输出格式异常       | 重试 1 次，仍异常则标记失败                                |
| 部署平台不可用           | 等待 5 分钟重试，3 次仍失败则暂停该项目                    |
| 单项目总耗时超过 72 小时 | 自动终止，归入封存阁，封存辞："超时了。也许这个想法太大了。" |
| 物帖池积压过多（>50）   | 展示"排队中"状态，按权重优先级依次进入造物流               |

## 编排层详细设计

### Cron 架构

开物局用 OpenClaw 的 **cron job** 驱动一切，不依赖 Gateway 内置心跳（可关闭）。三个独立的 cron job 各管各的事：

```
┌─────────────────────────────────────────────────────────┐
│  造物流更鼓（cron job #1）                                │
│                                                         │
│  类型：isolated session                                  │
│  间隔：按阶段配置（10-30 分钟）                           │
│  职责：推进当前造物令，控制造物流节奏                      │
│  绑定：编排层（不绑定特定 Agent）                         │
│                                                         │
│  cron 触发 → 编排层 tick() → 调用对应 Agent → 等产出     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  游商巡视（cron job #2）                                  │
│                                                         │
│  类型：isolated session                                  │
│  间隔：1-2 小时                                          │
│  职责：预采风、趋势监测、复盘验证                         │
│  绑定：agent: youshang                                   │
│                                                         │
│  cron 触发 → 游商独立会话 → 自由活动 → 产出写入日志/缓存 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  每日总结（cron job #3）                                  │
│                                                         │
│  类型：isolated session                                  │
│  间隔：每天 23:00                                        │
│  职责：回顾当天工作，提炼经验写入记忆文件                 │
│  绑定：编排层（不绑定特定 Agent）                         │
│                                                         │
│  cron 触发 → 查询今日数据 → LLM 提炼 → 写入 memory/     │
└─────────────────────────────────────────────────────────┘
```

#### 为什么用 cron 而不是 Gateway 内置心跳？

| | Gateway 心跳 | OpenClaw Cron |
|---|---|---|
| 时间精度 | 可能漂移 | 5 字段 cron 表达式，精确控制 |
| 持久化 | 随 Gateway 重启丢失 | 持久化在 `~/.openclaw/cron/jobs.json` |
| 会话隔离 | 共享主会话 | isolated session，每次独立，不污染记忆 |
| Agent 绑定 | 全局 | 可绑定特定 Agent（`--agent`） |
| 重试机制 | 无 | 自带指数退避（30s → 1m → 5m → 15m → 60m） |
| 可观测性 | 无日志 | 运行历史持久化在 `cron/runs/` |

**关键优势：isolated session**。每次 cron 触发创建独立会话，Agent 的主会话上下文不会被造物流任务污染。这从根本上解决了记忆混乱的问题。

#### Cron Job 配置

**造物流更鼓**：

```bash
openclaw cron add \
  --name "造物流更鼓" \
  --cron "*/20 * * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --message "执行造物流 tick：检查当前造物令状态，推进一步。" \
  --no-deliver
```

**游商巡视**：

```bash
openclaw cron add \
  --name "游商巡视" \
  --cron "0 */2 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --agent youshang \
  --message "自由活动：检查物帖池预采风、扫描行业趋势、回访已上线器物。" \
  --no-deliver
```

**游商巡视**：

```bash
openclaw cron add \
  --name "游商巡视" \
  --cron "0 */2 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --agent youshang \
  --message "自由活动：检查物帖池预采风、扫描行业趋势、回访已上线器物。" \
  --no-deliver
```

**每日总结**：

```bash
openclaw cron add \
  --name "每日总结" \
  --cron "0 23 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --message "回顾今天所有局中人的工作，提炼经验写入记忆文件。" \
  --no-deliver
```

> Gateway 内置心跳建议关闭（`heartbeat.every: null`），避免多余的唤醒。

#### Agent 的工作模式

| 模式 | 触发方 | 会话类型 | 产出去向 |
|---|---|---|---|
| 造物流任务 | 更鼓 cron → 编排层 tick | isolated（编排层创建） | phases.output，触发阶段流转 |
| 自由活动 | 巡视 cron → 直接唤醒 | isolated（cron 创建） | agent_logs / keywords.pre_scout_data |
| 每日总结 | 总结 cron → 编排层 | isolated（编排层创建） | memory/ 文件（不唤醒 Agent） |

两种模式的会话完全隔离，不存在记忆污染。

#### 哪些 Agent 可以自由活动？

| Agent | 可否自由活动 | 空闲时做什么 |
|---|---|---|
| 游商 | ✅ 可以 | 预采风、趋势监测、复盘验证 |
| 说客 | ❌ 不可以 | 过堂有严格轮次依赖，空闲时无事可做 |
| 诤臣 | ❌ 不可以 | 同上 |
| 掌秤 | ❌ 不可以 | 同上 |
| 画师 | ❌ 不可以 | 蓝图必须基于裁决结果 |
| 匠人 | ❌ 不可以 | 必须按蓝图执行，不能自由发挥 |
| 试剑 | ❌ 不可以 | 必须等锻造完成 |
| 鸣锣 | ❌ 不可以 | 必须等试剑通过 |

> MVP 阶段只有游商支持自由活动。后续可以考虑让诤臣在空闲时"巡视封存阁，回顾过往判断"，或让试剑"研究新的审查标准"——但这些不急。

#### 游商的自由活动

游商是唯一一个核心能力**不依赖造物流上游产出**的角色。他的采风能力随时可用。

空闲时游商可以做三件事：

**1. 预采风**

物帖池里排队的物帖，游商提前做轻量级市场扫描。等正式进入造物流时，已经有初步数据了，正式采风更快更准。

```
游商巡视 cron 触发（isolated session）
  │
  ├── 检查物帖池 → 有排队的物帖
  │
  ├── 取权重最高的物帖，做轻量级扫描
  │   （只搜索竞品数量和市场热度，不做完整的四维度分析）
  │
  ├── 结果写入 keywords.pre_scout_data（JSONB 字段）
  │   （不写入 phases.output，不触发阶段流转）
  │
  └── writeLog({ type: "free_activity", content: "顺路看了一眼「宠物心理咨询」，竞品不多，有意思。" })
```

**2. 趋势监测**

定期扫描行业热点，发现值得关注的新方向。产出是"游商见闻"，发布到展示网站的内坊板块。

```
游商巡视 cron 触发（isolated session）
  │
  ├── 无待预采风物帖
  │
  ├── 调用 web_search 扫描近期行业热点
  │
  ├── 生成"游商见闻"（一段简短的市场观察）
  │
  └── writeLog({ type: "insight", visibility: "public", content: "最近坊间都在聊 AI 编程助手，热度比上个月涨了三成。" })
```

**3. 复盘验证**

已鸣锣的器物，游商去看看市场反馈。补充到复盘数据里。

```
游商巡视 cron 触发（isolated session）
  │
  ├── 检查是否有已上线器物需要回访
  │   （launched_at 超过 7 天且未回访过）
  │
  ├── 搜索该器物的市场反馈、用户评价
  │
  └── 写入 retrospectives 的补充数据
```

#### 自由活动的安全边界

自由活动**绝对不能**做的事：

- ❌ 修改任何造物令的状态（projects / phases / tasks）
- ❌ 触发阶段流转
- ❌ 写入 phases.output

自由活动**可以**做的事：

- ✅ 写入 agent_logs（type: free_activity / insight）
- ✅ 写入 keywords.pre_scout_data（预采风缓存）
- ✅ 写入 retrospectives 的补充数据
- ✅ 更新自己的 MEMORY.md

#### 展示网站的体验提升

自由活动让展示网站在造物流空闲时也有内容更新：

```
造物流正在锻造中（匠人在干活，其他人闲着）
  │
  ├── 展示网站首页"此刻正在发生"：
  │   ├── 🔨 匠人正在完成「极简记账」的 T-005
  │   └── 🎒 游商顺路看了一眼物帖池里的「宠物心理咨询」
  │
  └── 内坊板块"游商见闻"：
      └── "最近坊间都在聊 AI 编程助手，热度比上个月涨了三成。"
```

#### 编排层 tick 流程

```
造物流更鼓 cron 触发（isolated session）
  │
  ├── 1. 检查 Gateway 健康状态
  │      └── 不健康 → 走 Gateway 故障恢复流程（见下方）
  │
  ├── 2. 取当前 running 的造物令（只有一个），计算下一步该做什么
  │      └── 如果没有 running 的造物令 → 检查物帖池，取权重最高的物帖立项
  │
  ├── 3. 通过 Gateway API 创建 isolated session，调用对应 Agent
  │      ├── 游商：采风任务
  │      ├── 说客：过堂第 2 轮，请反击
  │      └── 匠人：任务 T-003 待认领
  │
  ├── 4. 等待 Agent 产出（超时上限见异常兜底）
  │
  ├── 5. 收到产出后执行决策规则，更新数据库（事务内）
  │
  ├── 6. 发送 SSE 事件
  │
  └── 7. 如果阶段完成，下次 tick 自动进入下一阶段
```

> 注意：编排层 tick 不再写 HEARTBEAT.md。任务指令直接通过 cron 的 `--message` 或 Gateway API 的会话消息传递给 Agent。HEARTBEAT.md 仅保留为 Agent 的行为规则参考文档。

#### 编排层挂了会怎样？

```
造物流更鼓 cron 仍在触发，但 tick 逻辑无法执行
  │
  ├── 造物流暂停（cron 触发后发现无法连接编排层，记录错误日志）
  │
  ├── 游商巡视 cron 不受影响，继续自由活动
  │   （独立的 cron job，不依赖编排层）
  │
  ├── 展示网站看到：
  │   ├── "造物流暂停中"
  │   └── "游商仍在巡视坊间"（自由活动产出仍在更新）
  │
  └── 编排层恢复后 → 下次 cron 触发时正常 tick，从数据库中的当前状态继续
```

### 调度策略：单造物令排队

**同一时间只有一个造物令在造物流中运行。** 其余物帖在队列中排队等待。

#### 为什么排队而不是并行？

1. **注意力集中**：局中人把全部精力投入一个造物令，产出质量有保障
2. **叙事线清晰**：展示网站上访客跟着一个故事看到底，不会在多个项目间跳来跳去
3. **零风险**：不存在记忆污染、HEARTBEAT 覆盖、状态混乱等并发问题
4. **符合设定**：开物局是个小作坊，八个人围着一张物帖干活，这才是对的画面

#### 排队机制

```
物帖池（按权重排序）
  │
  ├── #1 宠物心理咨询  权重 87  ← 下一个进入造物流
  ├── #2 AI 算命       权重 72
  ├── #3 极简 CRM      权重 65
  └── ...

造物流（同一时间只有一个）
  │
  └── 当前：「极简记账」—— 锻造中，匠人正在完成 T-005

已完成
  ├── 器物坊：已鸣锣的器物
  └── 封存阁：被封存的物帖
```

#### 造物令完成后的衔接

```
当前造物令完成（鸣锣 or 封存）
  │
  ├── 1. 更新项目状态（launched / dead）
  ├── 2. 触发属性计算
  ├── 3. 所有局中人状态重置为 idle
  │
  ├── 4. 从物帖池取权重最高的物帖
  │      └── 如果池子空了 → 造物流空转，等待新物帖
  │
  ├── 5. 创建新的造物令（projects 记录）
  ├── 6. 进入采风阶段，下次心跳开始推进
  │
  └── 7. SSE 推送："新的造物令开始了"
```

#### 同一造物令内的串行/并行

造物令之间是排队的，但**同一个造物令内部**，锻造阶段的子任务仍然可以并行：

| 阶段 | 模式 | 原因 |
|---|---|---|
| 采风 | 串行 | 只有游商一人 |
| 过堂 | 严格串行 | 说客说完 → 诤臣反驳，有因果依赖 |
| 绘图 | 串行 | 只有画师一人 |
| 锻造 | **可并行** | 无依赖的子任务可同时分配给匠人的不同子角色 |
| 试剑 | 串行 | 只有试剑一人 |
| 鸣锣 | 串行 | 部署步骤有严格顺序 |

#### 过堂为什么必须串行？

```
❌ 错误：说客和诤臣同时发言
   说客："市场很大！"  ←── 同时 ──→  诤臣："风险很高！"
   （各说各的，没有交锋）

✅ 正确：说客先说，诤臣针对说客的论点反驳
   说客："市场很大，留存率只有 22%，说明现有产品不行"
       ↓
   诤臣："22% 的留存率恰恰说明用户不需要这类产品"
   （有来有回，有戏看）
```

#### 锻造的并行调度

```
心跳 N
  │
  ├── 检查任务依赖图
  ├── 找出所有 depends_on 已完成的 pending 任务
  ├── 按优先级排序，取前 3 个（并行上限）
  └── 分配给匠人的子角色：
      ├── 匠人·器（前端任务）
      ├── 匠人·骨（后端任务）
      └── 匠人·验（测试任务，需等被测代码完成）
```

匠人的三个子角色是**同一个 Agent 的三次独立调用**（不同 session），每次传入不同的任务和子角色指令。

#### 未来扩展：有限并行

MVP 阶段严格排队。如果后续需要提升吞吐量，可以放宽到"最多 2-3 个造物令同时运行"，但需要额外处理：

- 每个造物令绑定独立的 Agent session，防止上下文泄漏
- HEARTBEAT.md 改为按项目分区，或改用 Gateway API 直接传消息
- 局中人的 MEMORY.md 加项目标签，防止记忆污染
- 展示网站支持多条叙事线并行展示

这些改动不小，所以 MVP 先不做。

### 单次心跳的推进粒度

每个阶段在一次心跳中推进的"一步"定义不同：

| 阶段 | 一次心跳推进的内容 | 完成一个阶段需要的心跳数 |
|---|---|---|
| 采风 | 完成一个维度的调研（市场/用户/技术/商业） | 4-5 次 |
| 过堂 | 一轮完整辩论（说客发言 + 诤臣发言） | 3-5 次（含裁决） |
| 绘图 | 完成蓝图的一个部分（产品定义/架构/任务拆解） | 3-4 次 |
| 锻造 | 完成一个任务（可并行推进多个无依赖任务） | 取决于任务数 |
| 试剑 | 完成一个维度的审查（代码/安全/性能/功能/体验） | 5-6 次 |
| 鸣锣 | 完成部署清单的一个步骤 | 6-7 次 |

#### 过堂的心跳节奏

```
心跳 1 → 说客第 1 轮 + 诤臣第 1 轮（同一次心跳内顺序执行）
心跳 2 → 说客第 2 轮 + 诤臣第 2 轮
心跳 3 → 说客第 3 轮 + 诤臣第 3 轮
心跳 4 → （可选）说客第 4 轮 + 诤臣第 4 轮（分歧仍大时）
心跳 5 → 掌秤裁决
```

每次心跳推进一整轮（说客+诤臣各说一次），不是单个 Agent 的单次发言。这样过堂的节奏更紧凑，访客看到的是"一来一回"的完整交锋。

掌秤在最后一次心跳单独裁决，不和辩论混在一起——给裁决一个独立的"仪式感"。

### 阶段交接的原子性

阶段完成后的交接操作必须是原子的。使用数据库事务保证：

```
BEGIN TRANSACTION

  1. 更新 phases.status = 'completed'
  2. 更新 phases.output = { 产出数据 }
  3. 更新 phases.completed_at = now()
  4. 执行决策规则：
     ├── 通过 → 创建下一阶段的 phases 记录（status = 'pending'）
     │          更新 projects.current_phase
     └── 封存 → 更新 projects.status = 'dead'
  5. 写入 events 表

COMMIT

-- 事务提交后（非事务内）：
6. 更新对应 Agent 的 HEARTBEAT.md
7. 发送 SSE 事件
```

第 6、7 步在事务外执行。如果 HEARTBEAT.md 写入失败，下次心跳会重新检测到 pending 状态的阶段并重试。如果 SSE 推送失败，不影响数据一致性——展示网站下次轮询或重连时会拿到最新状态。

### Agent 执行中途崩溃

除了超时和格式异常，还需要处理以下情况：

| 异常 | 检测方式 | 处理 |
|---|---|---|
| Agent 会话被 Gateway 意外终止 | Gateway 返回 session_terminated 错误 | 重试 1 次，仍失败则标记阶段为 failed |
| Agent 写了日志但没提交产出 | 心跳检测到阶段 in_progress 但超过 2 个心跳周期无产出 | 标记为 stale，重试 1 次 |
| Agent 提交了产出但编排层没收到 | 编排层轮询 phases.output 是否已被 Agent 直接写入 | 如果 output 已有值，视为成功 |
| Gateway 重启导致所有会话丢失 | 编排层 tick 时检测 Gateway 健康状态 | 暂停所有项目，Gateway 恢复后从当前阶段重新开始 |

#### 阶段状态补充

在原有状态基础上增加 `stale`：

```
pending → in_progress → completed / failed / stale
                                              │
                                              └→ 重试 → in_progress（attempt + 1）
```

`stale` 表示"Agent 开始干了但没给结果"。编排层在下次心跳时自动重试 stale 的阶段。

### 封存辞的生成

| 封存原因 | 谁写封存辞 | 生成时机 |
|---|---|---|
| 采风评分 < 60 | 游商 | 采风完成的同一次心跳内，编排层额外调用游商生成一句封存辞 |
| 掌秤否决 | 掌秤 | 裁决的同一次心跳内，裁决书的 `epitaph` 字段就是封存辞 |
| 试剑 3 轮不通过 | 试剑 | 第 3 轮审查报告的同一次心跳内，编排层额外调用试剑生成封存辞 |
| 72 小时超时 | 编排层 | 自动生成固定文案："超时了。也许这个想法太大了。" |
| Agent 崩溃导致失败 | 编排层 | 自动生成："造物流中断。局中人遇到了意料之外的困难。" |

封存辞不需要额外的心跳，在封存决策的同一次 tick 内完成。

### 复盘触发机制

复盘由编排层的心跳驱动，不是独立的定时任务。

```
每次心跳 tick()
  │
  ├── 推进 running 项目（正常流程）
  │
  └── 检查复盘队列：
      │
      ├── 查询所有 status = 'launched' 的项目
      │
      ├── 对每个项目检查：
      │   ├── launched_at + 7天 已过 且 无 trigger_day=7 的复盘 → 触发 7 天复盘
      │   ├── launched_at + 30天 已过 且 无 trigger_day=30 的复盘 → 触发 30 天复盘
      │   └── launched_at + 90天 已过 且 无 trigger_day=90 的复盘 → 触发 90 天复盘
      │
      └── 触发复盘：
          1. 编排层收集器物的实际表现数据（访问量、留存率等）
          2. 编排层对比各局中人当初的判断 vs 实际结果
          3. 编排层计算属性变更
          4. 编排层生成复盘志摘要（或调用一个专门的 Agent 生成）
          5. 写入 retrospectives 表
          6. 更新 agent_stats 表
          7. 发送 SSE 事件 + 展示网站更新
```

复盘不需要唤醒局中人——它是编排层的**后台计算任务**，基于已有数据自动完成。如果未来想让复盘更有"人味"（比如让游商回顾自己的采风准不准），可以在复盘时额外唤醒对应 Agent 生成一段感想，但 MVP 阶段不需要。

### Gateway 故障恢复

```
编排层 tick()
  │
  ├── 1. 检查 Gateway 健康状态（GET /health）
  │
  ├── 健康 → 正常推进
  │
  └── 不健康 →
      ├── 标记所有 in_progress 的阶段为 stale
      ├── 记录 gateway_down 事件
      ├── SSE 推送"开物局暂时休息中"
      └── 后续每次 tick 继续检查，恢复后：
          ├── 标记 gateway_recovered 事件
          ├── 所有 stale 阶段自动重试
          └── SSE 推送"开物局恢复运转"
```

**断点续跑**：因为所有状态都持久化在数据库里（projects.status、phases.status、tasks.status），Gateway 恢复后编排层只需要遍历数据库，找到所有未完成的项目，从当前阶段继续推进。不需要额外的"恢复"逻辑——正常的 tick 流程本身就是幂等的。

### 心跳监控

编排层自身也需要被监控：

| 指标 | 告警条件 | 处理 |
|---|---|---|
| 心跳执行时间 | 单次 tick > 5 分钟 | 告警，检查是否有 Agent 卡住 |
| 心跳跳过 | 连续 3 次 tick 未执行 | 告警，检查编排层进程 |
| 项目停滞 | 某项目连续 10 次心跳无进展 | 告警，可能是 Agent 反复失败 |
| 封存率异常 | 连续 5 个项目都被封存 | 告警，检查 Agent prompt 或评分阈值 |

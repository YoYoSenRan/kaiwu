# 造物流 — 阶段设计

## 全景

```
物帖投入 → [采风] → [过堂] → [绘图] → [锻造] → [试剑] → [鸣锣] → 器物问世
            游商    说客/诤臣  画师     匠人     试剑     鸣锣
                    /掌秤
```

**全自动，无人工干预**。局中人自行决策，掌秤是整条造物流的"最高裁决者"。

## 推进策略：Cron 驱动

整条造物流由 OpenClaw 的 **cron job** 驱动。同一时间只有一个造物令在跑，其余物帖在池中排队。

```
Cron #1 · 造物流更鼓     每 20 分钟    推进当前造物令一步
Cron #2 · 游商巡视       每 2 小时     游商自由活动（预采风、趋势监测）
Cron #3 · 每日总结       每天 23:00    回顾当天工作，提炼经验
```

每次更鼓触发时，编排层取当前唯一的 running 造物令，推进一步：

```
更鼓触发
  │
  ▼
取当前 running 的造物令（同一时间只有一个）
  │
  ├── 无造物令在跑 → 从物帖池取权重最高的物帖，创建新造物令
  │
  ├── 有造物令 → 检查当前阶段
  │     │
  │     ├── 阶段进行中 → 推进一步（采风一个维度 / 过堂一轮 / 锻造一个区块 / ...）
  │     ├── 阶段刚完成 → 执行自动决策规则，决定进入下一阶段还是封存
  │     └── 阶段异常 → 重试或标记失败
  │
  ├── 写入事件日志
  │
  └── SSE 推送到展示网站
```

### 更鼓节奏配置

更鼓间隔可按阶段精细调控：

```bash
# 造物流更鼓（默认 20 分钟）
openclaw cron add --name "造物流更鼓" --cron "*/20 * * * *" ...
```

不同阶段可以用不同的 cron 间隔，或者在编排层 tick 内根据当前阶段动态决定是否跳过本次 tick：

| 阶段 | 建议间隔 | 理由 |
|---|---|---|
| 采风 | 20 分钟 | 默认节奏 |
| 过堂 | 10 分钟 | 辩论节奏快，更有张力 |
| 绘图 | 20 分钟 | 默认节奏 |
| 锻造 | 30 分钟 | 模拟写代码的节奏，每步更扎实 |
| 试剑 | 20 分钟 | 默认节奏 |
| 鸣锣 | 20 分钟 | 默认节奏 |

### 为什么用 Cron 驱动

1. **节奏完全可控**——cron 表达式调快调慢，不用改代码
2. **展示体验好**——访客每次来都能看到"又推进了一步"，而不是"已经跑完了"
3. **叙事线清晰**——一次只推进一个造物令，访客跟着一个故事看到底
4. **持久可靠**——cron 持久化在 Gateway，重启不丢，自带重试
5. **会话隔离**——每次 tick 是 isolated session，不污染 Agent 记忆

### 自动决策规则

| 衔接点           | 规则                                     |
| ---------------- | ---------------------------------------- |
| 探测完成         | 评分 ≥ 60 进入论证，< 60 归入封存阁        |
| 论证完成         | 掌秤裁决"通过"进入规划，"否决"归入封存阁 |
| 规划完成         | 自动进入构建                             |
| 构建完成         | 自动进入审查                             |
| 审查不通过       | 回退构建修复，最多 3 轮                  |
| 审查通过         | 自动进入发布                             |
| 发布冒烟测试失败 | 回滚，回退审查                           |

## Phase 1 — 采风（Scout）

**目标**：把一个简短的物帖扩展成完整的项目背景，并评估是否值得深入。

> 用户只提交关键词和简短理由（降低门槛），游商负责把它"充实"成一份完整的项目背景书。后续所有阶段——过堂、绘图、锻造——都基于这份背景书工作。

### 输入

- 物帖文本（1-20 字）
- 提交理由（20-200 字）
- 预采风数据（如果游商自由活动时已经扫描过，可选）

### Agent 工作内容

#### 第一步：项目背景生成

游商先把物帖扩展成一份**项目背景书**：

| 维度 | 内容 |
|---|---|
| 产品定位 | 这个器物是什么？一句话描述 |
| 目标用户 | 给谁用？用户画像（年龄、职业、场景） |
| 核心痛点 | 用户现在怎么解决这个问题？为什么不满意？ |
| 产品形态 | 这个器物长什么样？（网站 / App / 工具 / 插件） |
| 核心功能 | 3-5 个最重要的功能点 |
| 差异化 | 和现有竞品比，这个器物的独特之处是什么？ |

#### 第二步：市场调研

在项目背景的基础上做四维度调研：

| 维度       | 具体动作                               |
| ---------- | -------------------------------------- |
| 市场调研   | 搜索竞品、分析市场规模、趋势走向       |
| 用户需求   | 验证痛点是否真实、需求频次、付费意愿   |
| 技术可行性 | 作为前端展示页面的实现难度、视觉参考   |
| 差异化空间 | 现有竞品的弱点、可切入的角度           |

#### 第三步：综合评分

| 维度 | 分值 | 说明 |
|---|---|---|
| 市场规模 | 0-100 | 这个赛道有多大 |
| 竞争程度 | 0-100 | 竞品多不多、强不强 |
| 用户痛点 | 0-100 | 痛点是否真实、频繁 |
| 差异化空间 | 0-100 | 有没有独特的切入角度 |
| 展示潜力 | 0-100 | 做成展示页面能不能足够精美、有特色 |
| **综合评分** | 0-100 | 加权平均 |

> 注意"展示潜力"这个维度——因为我们的器物是前端展示页面，一个概念再好，如果做不出好看的页面，也不值得造。

### 输出

- 《采风报告》：包含项目背景书 + 四维度调研 + 综合评分
- 评分：综合分 + 各维度分
- 建议：🟢 推荐立项 / 🟡 需进一步论证 / 🔴 不建议
- 私货：游商的一句个人评价（可选）

### 采风报告的流转

采风报告是整个造物令的**基础文档**，后续每个阶段都会引用：

| 阶段 | 引用采风报告的哪部分 |
|---|---|
| 过堂 | 说客引用市场机会，诤臣引用竞争风险 |
| 绘图 | 画师基于"产品定位 + 核心功能 + 差异化"设计页面 |
| 锻造 | 匠人参考"产品形态 + 核心功能"理解要做什么 |
| 试剑 | 试剑对照"核心功能"验证完整性 |

### 自动决策规则

- 综合评分 ≥ 60 → 自动进入过堂阶段
- 综合评分 < 60 → 自动归入封存阁，生成封存辞
- 物帖过于模糊导致搜索无有效结果 → 标记为"无法评估"，归入封存阁

## Phase 2 — 论证（Council）

**目标**：多角度辩论，形成立项决策。掌秤是最终决策者。

### 参与 Agent

| 角色                    | 立场     | 关注点                             |
| ----------------------- | -------- | ---------------------------------- |
| 🟢 说客（Optimist）   | 支持做   | 市场机会、增长潜力、差异化空间     |
| 🔴 诤臣（Skeptic）    | 质疑做   | 风险、竞争壁垒、资源消耗、失败概率 |
| 🟡 掌秤（Pragmatist） | 最终仲裁 | 综合双方观点，做出最终裁决         |

### 辩论机制

1. **第一轮**：说客和诤臣各自基于采风报告发表初始观点
2. **第二轮**：针对对方观点进行反驳
3. **第三轮**：回应反驳，修正或坚持立场
4. **第四轮**（可选）：如果分歧仍然很大，进行最终陈述
5. **裁决**：掌秤综合所有论点，出具最终裁决

### 辩论规则

- 每轮发言必须引用具体数据或逻辑，禁止空洞断言
- 必须回应对方的核心论点，不能回避
- 每轮发言有字数上限，防止注水

### 输出

- 掌秤的最终裁决：✅ 通过（可附带条件）/ ❌ 否决
- 完整辩论记录（供展示网站回放）

### 自动决策规则

- 掌秤裁决"通过" → 自动进入规划阶段（附带的条件会传递给画师作为约束）
- 掌秤裁决"否决" → 自动归入封存阁，掌秤撰写封存辞

## Phase 3 — 绘图（Architect）

**目标**：将立项决策转化为一份精美前端页面的设计方案。

> 造物的目标不是"可用的 SaaS 产品"，而是**一个精美的前端展示页面**——像 Product Hunt 上的 Landing Page，内容丰富、视觉精美、有特色。不需要后端、数据库、用户认证。

### Agent 工作内容

1. **产品定位**
   - 一句话定位（这个器物是什么、给谁用、解决什么问题）
   - 核心卖点（3-5 个，用于页面展示）
   - 目标受众画像
   - 如果掌秤裁决附带条件，必须在设计中体现

2. **页面结构设计**
   - 页面分几个区块（Hero / 功能介绍 / 使用场景 / 定价 / FAQ / CTA）
   - 每个区块的内容大纲和文案方向
   - 页面的叙事逻辑（访客从上往下滚，看到什么、感受什么）

3. **视觉方向**
   - 配色方案（主色 + 辅色 + 背景色）
   - 字体选择
   - 视觉风格关键词（如"极简/温暖/专业/活泼"）
   - 参考网站或设计风格（给匠人一个明确的视觉锚点）

4. **交互设计**
   - 关键交互点（滚动动画、hover 效果、CTA 按钮）
   - 响应式断点策略
   - 特色交互（让这个页面与众不同的一个亮点）

5. **任务拆解**
   - 按页面区块拆成任务
   - 每个任务包含：区块名称、内容要求、视觉要求、交互要求
   - 任务之间的依赖关系（如 Hero 区先做，其他区块可并行）

### 输出

- 《造物蓝图》：包含上述所有内容
- 任务列表：自动初始化到造物流系统中

### 质量保障：蓝图的精细度

蓝图必须细到匠人**不需要做设计决策**——拿到任务就能直接写代码。

```
❌ 粗糙的蓝图：
   "做一个 Hero 区，展示产品名称和核心卖点"

✅ 精细的蓝图：
   Hero 区：
   - 布局：左文右图，桌面端 6:4 分栏，移动端上下堆叠
   - 标题："3 秒记一笔"，64px，font-bold，渐变色（#c23b22 → #d4833c）
   - 副标题："不需要学习，不需要分类，打开就记"，20px，text-muted
   - CTA 按钮："开始记账 →"，朱砂红底白字，hover 时微微放大
   - 右侧：产品截图 mockup，带手机边框，微微倾斜 3°
   - 背景：浅灰渐变，底部有一条细细的分隔线
   - 动画：标题从左侧 fadeIn，截图从右侧 slideIn，延迟 200ms
```

### 自动流转

- 蓝图完成后自动进入锻造阶段，任务列表分发给匠人

## Phase 4 — 锻造（Builder）

**目标**：根据蓝图，匠人分步骤实现一个精美的前端展示页面。不是一口气写完，而是**一个模块一个模块地打磨**，每一步都有过程感。

### 技术栈选型

技术栈由**画师在蓝图中推荐**，不硬性固定。但有一个推荐清单和基本约束：

**推荐清单**（画师优先从中选择）：

| 层面 | 推荐选项 | 说明 |
|---|---|---|
| 框架 | Next.js / Astro / Vite + React | 根据器物特点选择，静态展示优先 Astro |
| 样式 | UnoCSS / CSS Modules | UnoCSS 优先（preset-wind 兼容 Tailwind 类名），特殊视觉效果可用 CSS Modules |
| UI 组件 | shadcn/ui / Radix UI / 手写 | 有合适的预制组件就用，没有就手写 |
| 动画 | Framer Motion / CSS Animation / GSAP | 简单动画用 CSS，复杂交互用 Framer Motion 或 GSAP |
| 图标 | Lucide / Heroicons / 自定义 SVG | 根据视觉风格选择 |

**基本约束**（不可违反）：

- 必须是前端项目，可静态部署
- 必须支持响应式（桌面 / 平板 / 移动端）
- 部署目标：Vercel

### 代码规范

匠人写的每一行代码都必须遵守以下规范：

**目录结构**：

```
project-root/
├── src/
│   ├── app/                    # 页面入口
│   │   ├── layout.tsx          # 根布局（字体、meta、全局样式）
│   │   └── page.tsx            # 主页面（组装各区块组件）
│   ├── components/             # 组件目录
│   │   ├── sections/           # 页面区块组件（Hero、Features、Pricing...）
│   │   │   ├── Hero.tsx
│   │   │   ├── Features.tsx
│   │   │   └── ...
│   │   └── ui/                 # 通用 UI 组件（Button、Card、Badge...）
│   ├── lib/                    # 工具函数
│   │   └── utils.ts
│   └── styles/                 # 全局样式（如有）
├── public/                     # 静态资源（图片、图标、字体）
├── package.json
├── uno.config.ts               # UnoCSS 主题配置
├── tsconfig.json
└── .eslintrc.json
```

**编码风格**：

- TypeScript 严格模式，禁止 `any`
- 组件用 PascalCase 命名，文件名与组件名一致
- 每个区块组件独立一个文件，不超过 150 行
- 超过 150 行的组件必须拆分子组件
- className 合并用 `cn()` 工具函数
- 颜色、间距用 UnoCSS 语义 token，不硬编码

**注释规范**：

```tsx
/**
 * Hero 区块 — 首屏展示
 *
 * 布局：左文右图，桌面端 6:4 分栏
 * 动画：标题 fadeIn，截图 slideIn
 */
export function Hero() {
  // 渐变色标题需要特殊处理，bg-clip-text 在 Safari 有兼容问题
  const titleClass = cn(
    "text-5xl font-bold",
    "bg-gradient-to-r from-primary to-accent bg-clip-text text-transparent"
  )

  return (...)
}
```

- 每个区块组件顶部必须有 JSDoc 注释（说明布局和动画）
- 复杂逻辑加 Why 注释（为什么这样做，不是做了什么）
- 禁止无意义注释（`// 返回组件`）
- 禁止注释掉的死代码

**格式化和 Lint**：

- ESLint + Prettier，配置随项目模板提供
- 每次提交前必须通过 lint 检查
- 试剑阶段会验证 lint 零 error

### 分步锻造流程

匠人不是一口气写完所有代码。按**模块拆分、逐步实现、每步验证**的节奏推进：

```
Step 1 · 搭骨架
  │
  │  匠人·形：创建项目结构、根布局、空的区块组件文件
  │  产出：能跑起来的空页面，所有区块占位
  │
  │  ── 试剑轻检：项目能启动吗？目录结构对吗？ ──
  │
Step 2 · 做 Hero 区
  │
  │  匠人·形：Hero 区的 HTML 结构和布局
  │  匠人·色：Hero 区的配色、字体、间距
  │  匠人·动：Hero 区的入场动画
  │  产出：完整的 Hero 区块
  │
  │  ── 试剑轻检：Hero 视觉还原度？响应式？动画流畅度？ ──
  │
Step 3 · 做功能介绍区
  │
  │  匠人·形 + 匠人·色 + 匠人·动
  │  产出：完整的功能介绍区块
  │
  │  ── 试剑轻检 ──
  │
Step 4 · 做使用场景区
  │  ...（每个区块重复 Step 2-3 的模式）
  │
Step N · 整体打磨
  │
  │  匠人·色：全局色彩一致性检查、细节微调
  │  匠人·动：全页滚动体验、动画节奏调整
  │  产出：完整的页面
  │
  │  ── 试剑全检：完整审查 ──
```

**每一步都是一次更鼓**。访客在展示网站上能看到：

```
🔨 匠人刚完成「极简记账」的 Hero 区，标题用了朱砂红渐变
🔨 匠人正在做「极简记账」的功能介绍区，三列卡片布局
🗡 试剑看了一眼 Hero 区："配色不错，但移动端标题太大了"
🔨 匠人调小了移动端标题字号，从 48px 改到 36px
```

### 试剑的分步介入

试剑不是最后才出场。每完成一个区块，试剑做一次**轻检**（只看当前区块），全部完成后做一次**全检**（看整体）。

| 检查类型 | 时机 | 范围 | 严格度 |
|---|---|---|---|
| 轻检 | 每个区块完成后 | 只看当前区块 | 🟡 宜修 + 🔴 必修 |
| 全检 | 所有区块完成后 | 整个页面 | 完整审查清单 |

轻检发现的问题**当场修复**，不走回炉流程。只有全检不通过才触发正式回炉。

这样的好处：
- 问题早发现早修复，不会积累到最后
- 匠人和试剑形成"做一块、查一块"的协作节奏
- 展示网站上能看到两人的互动（"试剑说移动端标题太大了" → "匠人改了"）

### 内容生成

页面上的文案、数据、截图不是匠人写的——是**画师在蓝图中定义的**。匠人只负责把内容"排版"到页面上。

| 内容类型 | 来源 | 示例 |
|---|---|---|
| 标题和文案 | 画师蓝图 | "3 秒记一笔" |
| 功能描述 | 画师蓝图 | "自动分类、智能提醒、月度报告" |
| 产品截图 | AI 生成（编排层调用图像生成 API） | 产品界面 mockup |
| 图标 | 画师蓝图指定的图标库 | Lucide / Heroicons / 自定义 SVG |
| 数据/数字 | 画师蓝图（虚构但合理） | "已有 10,000+ 用户" |

### 产出目录规范

造物令的产出分两个地方存放，各司其职：

#### 结构化数据 → 数据库

| 产出 | 存储位置 | 用途 |
|---|---|---|
| 采风报告 | phases.output（scout） | 展示网站展示、过堂引用 |
| 辩论记录 | debates 表 | 过堂回放、对话流 |
| 裁决书 | phases.output（council） | 展示网站展示 |
| 蓝图 | phases.output（architect） | 匠人参照、展示网站展示 |
| 试剑报告 | phases.output（inspector） | 展示网站展示 |
| 鸣锣报告 | phases.output（deployer） | 展示网站展示 |
| 过程日志 | agent_logs 表 | 对话流、群聊记录 |
| 事件 | events 表 | 时间线、SSE 推送 |

#### 器物代码 → 文件系统

每个造物令的代码放在**独立的目录**中，不放在 Agent 的 workspace 里。

```
~/.openclaw/products/
├── {project-slug}/                  # 每个造物令一个目录（如 jinji-jizhang）
│   ├── .kaiwu/                      # 开物局元数据（不部署）
│   │   ├── blueprint.json           # 蓝图快照（从 phases.output 复制，匠人离线参考用）
│   │   ├── tasks.json               # 任务清单和状态
│   │   └── reviews/                 # 试剑轻检记录
│   │       ├── step-01-skeleton.md
│   │       ├── step-02-hero.md
│   │       └── ...
│   │
│   ├── src/                         # 器物源码（匠人在这里写代码）
│   │   ├── app/
│   │   │   ├── layout.tsx
│   │   │   └── page.tsx
│   │   ├── components/
│   │   │   ├── sections/            # 页面区块组件
│   │   │   └── ui/                  # 通用 UI 组件
│   │   ├── lib/
│   │   └── styles/
│   │
│   ├── public/                      # 静态资源
│   │   ├── images/
│   │   └── fonts/
│   │
│   ├── package.json
│   ├── uno.config.ts
│   ├── tsconfig.json
│   ├── .eslintrc.json
│   └── README.md                    # 器物说明（鸣锣后自动生成）
│
├── {another-project-slug}/
└── ...
```

**关键规则**：

- **一个造物令一个目录**，用 `project.slug` 命名（如 `jinji-jizhang`）
- **`.kaiwu/` 目录**存放开物局的元数据（蓝图快照、任务清单、轻检记录），不会被部署
- **`src/` 目录**是器物的实际代码，匠人只在这里写代码
- **Agent 的 workspace 里不放任何项目代码**——workspace 是"住所"，products 是"工坊"
- 造物令封存时，目录保留但标记为 archived（不删除，留作记忆和展示）
- 鸣锣后，整个目录推送到 Git 仓库并部署到 Vercel

#### 编排层如何告诉 Agent 在哪里写代码？

编排层在调用匠人时，通过 cron message 传入项目路径：

```
task: 完成 Hero 区的结构和布局
projectPath: ~/.openclaw/products/jinji-jizhang
section: hero
blueprint: （蓝图中 Hero 区的详细描述）
```

匠人的 tool（code_gen / file_ops）操作的目标目录就是这个 `projectPath`，不是自己的 workspace。

#### 为什么不放在 Agent workspace 里？

- workspace 是 Agent 的"灵魂"（SOUL.md、MEMORY.md），混入项目代码会污染
- 多个造物令的代码如果都在 workspace 里，目录会越来越乱
- products 目录独立于任何 Agent，试剑、鸣锣都能访问同一个目录
- 造物令封存后代码仍然保留，不影响 Agent 的 workspace 清洁

### 自动流转

- 所有区块完成 + 试剑全检通过 → 自动进入鸣锣阶段

## Phase 5 — 试剑（Inspector）

**目标**：确保器物（前端展示页面）的视觉质量和体验达标。试剑在锻造过程中就开始介入（轻检），全部完成后做最终审查（全检）。

### 轻检（每个区块完成后）

快速检查当前区块，不走正式流程：

| 维度 | 检查内容 |
|---|---|
| 蓝图还原 | 布局、配色、字体是否与蓝图一致 |
| 响应式 | 桌面/平板/移动端三个断点是否正常 |
| 内容完整 | 文案、图片是否齐全，无占位符 |
| 动画 | 交互动画是否流畅，时机是否恰当 |

轻检结果直接反馈给匠人，当场修复。不写入正式的试剑报告。

### 全检（所有区块完成后）

完整的审查，走正式流程：

| 维度 | 内容 | 标准 |
|---|---|---|
| 视觉还原 | 整体页面是否符合蓝图的设计方案 | 配色、布局、字体与蓝图一致 |
| 内容完整 | 所有区块的文案、图片是否齐全 | 无占位符、无 lorem ipsum、无空白区域 |
| 响应式 | 桌面/平板/移动端三个断点 | 无溢出、无错位、可读性好 |
| 动画体验 | 全页滚动体验、交互动画 | 无卡顿、无闪烁、节奏自然 |
| 加载性能 | 首屏加载速度 | Lighthouse Performance ≥ 90 |
| 特色亮点 | 是否有让人眼前一亮的设计 | 至少一个"哇"的瞬间 |
| 代码规范 | Lint 通过、目录结构、注释 | ESLint 零 error，组件拆分合理，注释规范 |
| 整体印象 | 作为一个访客，第一眼感受 | 精美、专业、有特色，不粗糙 |

### 输出

- 《试剑报告》：通过 / 不通过
- 问题清单：按严重程度排序

### 自动决策规则

- 🔴 严重问题 = 0 且 🟡 一般问题 ≤ 3 → 通过，自动进入鸣锣
- 存在 🔴 严重问题 → 不通过，自动回退到锻造阶段修复
- 回退修复后重新试剑，最多 3 轮。3 轮仍不通过 → 标记为"锻造失败"，归入封存阁

## 编排层详细设计

### Cron 架构

开物局用 OpenClaw 的 **cron job** 驱动一切，不依赖 Gateway 内置心跳（可关闭）。三个独立的 cron job 各管各的事：

```
┌─────────────────────────────────────────────────────────┐
│  造物流更鼓（cron job #1）                                │
│                                                         │
│  类型：isolated session                                  │
│  间隔：按阶段配置（10-30 分钟）                           │
│  职责：推进当前造物令，控制造物流节奏                      │
│  绑定：编排层（不绑定特定 Agent）                         │
│                                                         │
│  cron 触发 → 编排层 tick() → 调用对应 Agent → 等产出     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  游商巡视（cron job #2）                                  │
│                                                         │
│  类型：isolated session                                  │
│  间隔：1-2 小时                                          │
│  职责：预采风、趋势监测、复盘验证                         │
│  绑定：agent: youshang                                   │
│                                                         │
│  cron 触发 → 游商独立会话 → 自由活动 → 产出写入日志/缓存 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  每日总结（cron job #3）                                  │
│                                                         │
│  类型：isolated session                                  │
│  间隔：每天 23:00                                        │
│  职责：回顾当天工作，提炼经验写入记忆文件                 │
│  绑定：编排层（不绑定特定 Agent）                         │
│                                                         │
│  cron 触发 → 查询今日数据 → LLM 提炼 → 写入 memory/     │
└─────────────────────────────────────────────────────────┘
```

#### 为什么用 cron 而不是 Gateway 内置心跳？

| | Gateway 心跳 | OpenClaw Cron |
|---|---|---|
| 时间精度 | 可能漂移 | 5 字段 cron 表达式，精确控制 |
| 持久化 | 随 Gateway 重启丢失 | 持久化在 `~/.openclaw/cron/jobs.json` |
| 会话隔离 | 共享主会话 | isolated session，每次独立，不污染记忆 |
| Agent 绑定 | 全局 | 可绑定特定 Agent（`--agent`） |
| 重试机制 | 无 | 自带指数退避（30s → 1m → 5m → 15m → 60m） |
| 可观测性 | 无日志 | 运行历史持久化在 `cron/runs/` |

**关键优势：isolated session**。每次 cron 触发创建独立会话，Agent 的主会话上下文不会被造物流任务污染。这从根本上解决了记忆混乱的问题。

#### Cron Job 配置

**造物流更鼓**：

```bash
openclaw cron add \
  --name "造物流更鼓" \
  --cron "*/20 * * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --message "执行造物流 tick：检查当前造物令状态，推进一步。" \
  --no-deliver
```

**游商巡视**：

```bash
openclaw cron add \
  --name "游商巡视" \
  --cron "0 */2 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --agent youshang \
  --message "自由活动：检查物帖池预采风、扫描行业趋势、回访已上线器物。" \
  --no-deliver
```

**游商巡视**：

```bash
openclaw cron add \
  --name "游商巡视" \
  --cron "0 */2 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --agent youshang \
  --message "自由活动：检查物帖池预采风、扫描行业趋势、回访已上线器物。" \
  --no-deliver
```

**每日总结**：

```bash
openclaw cron add \
  --name "每日总结" \
  --cron "0 23 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --message "回顾今天所有局中人的工作，提炼经验写入记忆文件。" \
  --no-deliver
```

> Gateway 内置心跳建议关闭（`heartbeat.every: null`），避免多余的唤醒。

#### Agent 的工作模式

| 模式 | 触发方 | 会话类型 | 产出去向 |
|---|---|---|---|
| 造物流任务 | 更鼓 cron → 编排层 tick | isolated（编排层创建） | phases.output，触发阶段流转 |
| 自由活动 | 巡视 cron → 直接唤醒 | isolated（cron 创建） | agent_logs / keywords.pre_scout_data |
| 每日总结 | 总结 cron → 编排层 | isolated（编排层创建） | memory/ 文件（不唤醒 Agent） |

两种模式的会话完全隔离，不存在记忆污染。

#### 哪些 Agent 可以自由活动？

| Agent | 可否自由活动 | 空闲时做什么 |
|---|---|---|
| 游商 | ✅ 可以 | 预采风、趋势监测、复盘验证 |
| 说客 | ❌ 不可以 | 过堂有严格轮次依赖，空闲时无事可做 |
| 诤臣 | ❌ 不可以 | 同上 |
| 掌秤 | ❌ 不可以 | 同上 |
| 画师 | ❌ 不可以 | 蓝图必须基于裁决结果 |
| 匠人 | ❌ 不可以 | 必须按蓝图执行，不能自由发挥 |
| 试剑 | ❌ 不可以 | 必须等锻造完成 |
| 鸣锣 | ❌ 不可以 | 必须等试剑通过 |

> MVP 阶段只有游商支持自由活动。后续可以考虑让诤臣在空闲时"巡视封存阁，回顾过往判断"，或让试剑"研究新的审查标准"——但这些不急。

#### 游商的自由活动

游商是唯一一个核心能力**不依赖造物流上游产出**的角色。他的采风能力随时可用。

空闲时游商可以做三件事：

**1. 预采风**

物帖池里排队的物帖，游商提前做轻量级市场扫描。等正式进入造物流时，已经有初步数据了，正式采风更快更准。

```
游商巡视 cron 触发（isolated session）
  │
  ├── 检查物帖池 → 有排队的物帖
  │
  ├── 取权重最高的物帖，做轻量级扫描
  │   （只搜索竞品数量和市场热度，不做完整的四维度分析）
  │
  ├── 结果写入 keywords.pre_scout_data（JSONB 字段）
  │   （不写入 phases.output，不触发阶段流转）
  │
  └── writeLog({ type: "free_activity", content: "顺路看了一眼「宠物心理咨询」，竞品不多，有意思。" })
```

**2. 趋势监测**

定期扫描行业热点，发现值得关注的新方向。产出是"游商见闻"，发布到展示网站的内坊板块。

```
游商巡视 cron 触发（isolated session）
  │
  ├── 无待预采风物帖
  │
  ├── 调用 web_search 扫描近期行业热点
  │
  ├── 生成"游商见闻"（一段简短的市场观察）
  │
  └── writeLog({ type: "insight", visibility: "public", content: "最近坊间都在聊 AI 编程助手，热度比上个月涨了三成。" })
```

**3. 复盘验证**

已鸣锣的器物，游商去看看市场反馈。补充到复盘数据里。

```
游商巡视 cron 触发（isolated session）
  │
  ├── 检查是否有已上线器物需要回访
  │   （launched_at 超过 7 天且未回访过）
  │
  ├── 搜索该器物的市场反馈、用户评价
  │
  └── 写入 retrospectives 的补充数据
```

#### 自由活动的安全边界

自由活动**绝对不能**做的事：

- ❌ 修改任何造物令的状态（projects / phases / tasks）
- ❌ 触发阶段流转
- ❌ 写入 phases.output

自由活动**可以**做的事：

- ✅ 写入 agent_logs（type: free_activity / insight）
- ✅ 写入 keywords.pre_scout_data（预采风缓存）
- ✅ 写入 retrospectives 的补充数据
- ✅ 更新自己的 MEMORY.md

#### 展示网站的体验提升

自由活动让展示网站在造物流空闲时也有内容更新：

```
造物流正在锻造中（匠人在干活，其他人闲着）
  │
  ├── 展示网站首页"此刻正在发生"：
  │   ├── 🔨 匠人正在完成「极简记账」的 T-005
  │   └── 🎒 游商顺路看了一眼物帖池里的「宠物心理咨询」
  │
  └── 内坊板块"游商见闻"：
      └── "最近坊间都在聊 AI 编程助手，热度比上个月涨了三成。"
```

#### 编排层 tick 流程

```
造物流更鼓 cron 触发（isolated session）
  │
  ├── 1. 检查 Gateway 健康状态
  │      └── 不健康 → 走 Gateway 故障恢复流程（见下方）
  │
  ├── 2. 取当前 running 的造物令（只有一个），计算下一步该做什么
  │      └── 如果没有 running 的造物令 → 检查物帖池，取权重最高的物帖立项
  │
  ├── 3. 通过 Gateway API 创建 isolated session，调用对应 Agent
  │      ├── 游商：采风任务
  │      ├── 说客：过堂第 2 轮，请反击
  │      └── 匠人：任务 T-003 待认领
  │
  ├── 4. 等待 Agent 产出（超时上限见异常兜底）
  │
  ├── 5. 收到产出后执行决策规则，更新数据库（事务内）
  │
  ├── 6. 发送 SSE 事件
  │
  └── 7. 如果阶段完成，下次 tick 自动进入下一阶段
```

> 注意：编排层 tick 不再写 HEARTBEAT.md。任务指令直接通过 cron 的 `--message` 或 Gateway API 的会话消息传递给 Agent。HEARTBEAT.md 仅保留为 Agent 的行为规则参考文档。

#### 编排层挂了会怎样？

```
造物流更鼓 cron 仍在触发，但 tick 逻辑无法执行
  │
  ├── 造物流暂停（cron 触发后发现无法连接编排层，记录错误日志）
  │
  ├── 游商巡视 cron 不受影响，继续自由活动
  │   （独立的 cron job，不依赖编排层）
  │
  ├── 展示网站看到：
  │   ├── "造物流暂停中"
  │   └── "游商仍在巡视坊间"（自由活动产出仍在更新）
  │
  └── 编排层恢复后 → 下次 cron 触发时正常 tick，从数据库中的当前状态继续
```

### 调度策略：单造物令排队

**同一时间只有一个造物令在造物流中运行。** 其余物帖在队列中排队等待。

#### 为什么排队而不是并行？

1. **注意力集中**：局中人把全部精力投入一个造物令，产出质量有保障
2. **叙事线清晰**：展示网站上访客跟着一个故事看到底，不会在多个项目间跳来跳去
3. **零风险**：不存在记忆污染、HEARTBEAT 覆盖、状态混乱等并发问题
4. **符合设定**：开物局是个小作坊，八个人围着一张物帖干活，这才是对的画面

#### 排队机制

```
物帖池（按权重排序）
  │
  ├── #1 宠物心理咨询  权重 87  ← 下一个进入造物流
  ├── #2 AI 算命       权重 72
  ├── #3 极简 CRM      权重 65
  └── ...

造物流（同一时间只有一个）
  │
  └── 当前：「极简记账」—— 锻造中，匠人正在完成 T-005

已完成
  ├── 器物坊：已鸣锣的器物
  └── 封存阁：被封存的物帖
```

#### 造物令完成后的衔接

```
当前造物令完成（鸣锣 or 封存）
  │
  ├── 1. 更新项目状态（launched / dead）
  ├── 2. 触发属性计算
  ├── 3. 所有局中人状态重置为 idle
  │
  ├── 4. 从物帖池取权重最高的物帖
  │      └── 如果池子空了 → 造物流空转，等待新物帖
  │
  ├── 5. 创建新的造物令（projects 记录）
  ├── 6. 进入采风阶段，下次更鼓开始推进
  │
  └── 7. SSE 推送："新的造物令开始了"
```

#### 同一造物令内的串行/并行

造物令之间是排队的，但**同一个造物令内部**，锻造阶段的子任务仍然可以并行：

| 阶段 | 模式 | 原因 |
|---|---|---|
| 采风 | 串行 | 只有游商一人 |
| 过堂 | 严格串行 | 说客说完 → 诤臣反驳，有因果依赖 |
| 绘图 | 串行 | 只有画师一人 |
| 锻造 | **可并行** | 无依赖的子任务可同时分配给匠人的不同子角色 |
| 试剑 | 串行 | 只有试剑一人 |
| 鸣锣 | 串行 | 部署步骤有严格顺序 |

#### 过堂为什么必须串行？

```
❌ 错误：说客和诤臣同时发言
   说客："市场很大！"  ←── 同时 ──→  诤臣："风险很高！"
   （各说各的，没有交锋）

✅ 正确：说客先说，诤臣针对说客的论点反驳
   说客："市场很大，留存率只有 22%，说明现有产品不行"
       ↓
   诤臣："22% 的留存率恰恰说明用户不需要这类产品"
   （有来有回，有戏看）
```

#### 锻造的并行调度

```
更鼓 N
  │
  ├── 检查任务依赖图
  ├── 找出所有 depends_on 已完成的 pending 任务
  ├── 按优先级排序，取前 3 个（并行上限）
  └── 分配给匠人的子角色：
      ├── 匠人·形（结构和布局任务）
      ├── 匠人·色（视觉和样式任务）
      └── 匠人·动（交互和动画任务，需等骨架完成）
```

匠人的三个子角色是**同一个 Agent 的三次独立调用**（不同 session），每次传入不同的任务和子角色指令。

#### 未来扩展：有限并行

MVP 阶段严格排队。如果后续需要提升吞吐量，可以放宽到"最多 2-3 个造物令同时运行"，但需要额外处理：

- 每个造物令绑定独立的 Agent session，防止上下文泄漏
- HEARTBEAT.md 改为按项目分区，或改用 Gateway API 直接传消息
- 局中人的 MEMORY.md 加项目标签，防止记忆污染
- 展示网站支持多条叙事线并行展示

这些改动不小，所以 MVP 先不做。

### 单次更鼓的推进粒度

每个阶段在一次更鼓中推进的"一步"定义不同：

| 阶段 | 一次更鼓推进的内容 | 完成一个阶段需要的更鼓数 |
|---|---|---|
| 采风 | 完成一个维度的调研（市场/用户/技术/商业） | 4-5 次 |
| 过堂 | 一轮完整辩论（说客发言 + 诤臣发言） | 3-5 次（含裁决） |
| 绘图 | 完成蓝图的一个部分（产品定义/架构/任务拆解） | 3-4 次 |
| 锻造 | 完成一个任务（可并行推进多个无依赖任务） | 取决于任务数 |
| 试剑 | 完成一个维度的审查（代码/安全/性能/功能/体验） | 5-6 次 |
| 鸣锣 | 完成部署清单的一个步骤 | 6-7 次 |

#### 过堂的更鼓节奏

```
更鼓 1 → 说客第 1 轮 + 诤臣第 1 轮（同一次更鼓内顺序执行）
更鼓 2 → 说客第 2 轮 + 诤臣第 2 轮
更鼓 3 → 说客第 3 轮 + 诤臣第 3 轮
更鼓 4 → （可选）说客第 4 轮 + 诤臣第 4 轮（分歧仍大时）
更鼓 5 → 掌秤裁决
```

每次更鼓推进一整轮（说客+诤臣各说一次），不是单个 Agent 的单次发言。这样过堂的节奏更紧凑，访客看到的是"一来一回"的完整交锋。

掌秤在最后一次更鼓单独裁决，不和辩论混在一起——给裁决一个独立的"仪式感"。

### 阶段交接的原子性

阶段完成后的交接操作必须是原子的。使用数据库事务保证：

```
BEGIN TRANSACTION

  1. 更新 phases.status = 'completed'
  2. 更新 phases.output = { 产出数据 }
  3. 更新 phases.completed_at = now()
  4. 执行决策规则：
     ├── 通过 → 创建下一阶段的 phases 记录（status = 'pending'）
     │          更新 projects.current_phase
     └── 封存 → 更新 projects.status = 'dead'
  5. 写入 events 表

COMMIT

-- 事务提交后（非事务内）：
6. 更新对应 Agent 的 HEARTBEAT.md
7. 发送 SSE 事件
```

第 6、7 步在事务外执行。如果写入失败，下次更鼓会重新检测到 pending 状态的阶段并重试。如果 SSE 推送失败，不影响数据一致性——展示网站下次轮询或重连时会拿到最新状态。

### 异常处理与渐进式自愈

当某个阶段的 Agent 执行失败或无进展时，编排层不是简单地"重试 → 放弃"，而是**逐级升级处理策略**，同时用指数退避减少无效消耗。

#### 四级响应机制

```
第 1 次失败 → L1 立即重试（可能是偶发问题）
第 2 次失败 → L2 调整重试（换个方式再来）
第 3 次失败 → L3 降速观察（拉长更鼓间隔，减少浪费）
第 4 次失败 → L4 暂停造物令（标记 blocked，等待恢复）
```

| 级别 | 触发条件 | 处理方式 | 更鼓间隔 | 展示网站显示 |
|---|---|---|---|---|
| L1 · 重试 | 第 1 次失败 | 同样的指令再调用一次 Agent | 不变 | "局中人遇到了点小问题，正在重试" |
| L2 · 调整重试 | 连续 2 次失败 | prompt 中追加上次错误信息，让 Agent 换个方式 | 不变 | "局中人调整了策略，再试一次" |
| L3 · 降速 | 连续 3 次失败 | 更鼓间隔翻倍，减少无效 token 消耗 | ×2（上限 120min） | "造物流放慢了脚步，局中人在想办法" |
| L4 · 暂停 | 连续 4 次失败 | 造物令标记为 blocked，停止推进 | 停止 | "造物流暂停了。局中人遇到了难题。" |

#### L2 调整重试

不是盲目重复，而是在 prompt 中加入**上次失败的上下文**：

```
原始指令：
  "为「极简记账」采风，完成市场调研维度"

L2 调整后：
  "为「极简记账」采风，完成市场调研维度。
   上次尝试失败了，原因是：web_search 返回结果为空。
   请换一个搜索角度，比如用英文关键词或者搜索竞品名称。"
```

#### L3 指数退避

```
正常间隔：20 分钟
第 3 次失败后：40 分钟
仍失败：80 分钟
上限：120 分钟（不再翻倍）

恢复：任何一次成功 → 立即恢复正常间隔，失败计数归零
```

#### L4 暂停后的恢复

```
造物令标记为 blocked
  │
  ├── 自动恢复：
  │   ├── 每小时检查 Gateway 健康状态
  │   ├── Gateway 恢复 → 解除 blocked，从当前阶段重新开始（失败计数归零）
  │   └── 连续 24 小时仍 blocked → 自动封存
  │       封存辞："造物流中断。局中人遇到了意料之外的困难。"
  │
  └── 展示网站：
      ├── 造物坊看板标记"暂停中"
      └── 群聊记录："造物流遇到了困难，暂时停下来想想办法"
```

#### 各类异常的检测和初始级别

| 异常类型 | 检测方式 | 初始级别 |
|---|---|---|
| Agent 调用超时 | Gateway API 返回 timeout | L1 |
| Agent 输出格式异常 | JSON 解析失败或缺少必要字段 | L2 |
| Agent 返回空内容 | output 为空或只有模板文字 | L2 |
| Agent 会话被终止 | Gateway 返回 session_terminated | L1 |
| Agent 无响应（stale） | 超过 2 个更鼓周期无产出 | L2 |
| Gateway 不可用 | health check 失败 | L4（直接暂停） |
| 部署平台不可用 | Vercel API 返回错误 | L3（降速等恢复） |

#### 阶段状态

```
pending → in_progress → completed / failed / stale / blocked
                          │         │        │        │
                          │         │        │        └→ 自动恢复或 24h 后封存
                          │         │        └→ L2 调整重试
                          │         └→ L1 重试
                          └→ 成功，失败计数归零
```

### 封存辞的生成

| 封存原因 | 谁写封存辞 | 生成时机 |
|---|---|---|
| 采风评分 < 60 | 游商 | 采风完成的同一次 tick 内 |
| 掌秤否决 | 掌秤 | 裁决书的 `epitaph` 字段 |
| 试剑 3 轮不通过 | 试剑 | 第 3 轮审查的同一次 tick 内 |
| 72 小时超时 | 编排层 | "超时了。也许这个想法太大了。" |
| L4 暂停 24h 未恢复 | 编排层 | "造物流中断。局中人遇到了意料之外的困难。" |

封存辞在封存决策的同一次 tick 内完成，不需要额外的更鼓。

### 复盘触发机制

复盘由编排层的更鼓 cron 驱动，不是独立的定时任务。

```
每次更鼓 tick()
  │
  ├── 推进 running 造物令（正常流程）
  │
  └── 检查复盘队列：
      │
      ├── 查询所有 status = 'launched' 的项目
      │
      ├── 对每个项目检查：
      │   ├── launched_at + 7天 已过 且 无 trigger_day=7 的复盘 → 触发 7 天复盘
      │   ├── launched_at + 30天 已过 且 无 trigger_day=30 的复盘 → 触发 30 天复盘
      │   └── launched_at + 90天 已过 且 无 trigger_day=90 的复盘 → 触发 90 天复盘
      │
      └── 触发复盘：
          1. 编排层收集器物的实际表现数据
          2. 编排层对比各局中人当初的判断 vs 实际结果
          3. ���排层计算属性变更
          4. 编排层生成复盘志摘要（或调用一个专门的 Agent 生成）
          5. 写入 retrospectives 表
          6. 更新 agent_stats 表
          7. 发送 SSE 事件 + 展示网站更新
```

#### 复盘数据从哪来？

| 数据 | 来源 | 获取方式 |
|---|---|---|
| 页面访问量 | 器物部署平台（Vercel Analytics） | Vercel API 定时拉取 |
| 可用性（uptime） | 鸣锣阶段接入的监控（如 UptimeRobot） | 监控 API 拉取 |
| 用户反馈 | 游商自由活动中的回访（web_search） | 搜索器物名称 + "评价/反馈" |
| 是否仍在运行 | 编排层直接访问器物 URL | HTTP HEAD 请求，检查状态码 |

MVP 阶段只用**最简单的两个指标**：
- **是否仍在运行**：编排层 HEAD 请求器物 URL，200 = 活着，其他 = 挂了
- **游商回访**：游商自由活动中搜索器物的市场反馈（已在记忆系统设计中定义）

后续可以接入 Vercel Analytics API 获取更精确的数据，但 MVP 不依赖它。

#### 复盘对比维度

| 角色 | 当初的判断（从 phases.output 提取） | 对比什么 |
|---|---|---|
| 游商 | 采风评分和各维度判断 | 评分 vs 器物实际表现（还活着？有人用？） |
| 说客 | 支持的核心论点 | 论点是否被市场验证 |
| 诤臣 | 质疑的核心风险 | 风险是否出现 |
| 掌秤 | 裁决和附带条件 | 条件是否合理 |
| 画师 | 蓝图的设计方案 | 设计是否合适 |
| 匠人 | 实际工时和困难 | 预估 vs 实际的偏差 |
| 试剑 | 审查发现的问题 | 放行后有没有漏掉的问题 |
| 鸣锣 | 部署方案 | 有没有需要回滚的情况 |

#### 复盘失败的降级

如果无法获取器物的实际表现数据（比如器物 URL 已失效）：
- 跳过"实际表现"维度的对比
- 仍然执行"局中人判断一致性"的对比
- 在复盘志中标注"部分数据缺失"
- 属性更新只基于可获取的数据

复盘不需要唤醒局中人——它是编排层的**后台计算任务**。MVP 阶段基于已有数据自动完成。

### Gateway 故障恢复

Gateway 不可用时直接进入 L4 暂停：

```
编排层 tick()
  │
  ├── 1. 检查 Gateway 健康状态（GET /health）
  │
  ├── 健康 → 正常推进
  │
  └── 不健康 →
      ├── 造物令标记为 blocked（L4）
      ├── 记录 gateway_down 事件
      ├── SSE 推送"开物局暂时休息中"
      └── 后续每小时检查，恢复后：
          ├── 解除 blocked，失败计数归零
          ├── 记录 gateway_recovered 事件
          └── SSE 推送"开物局恢复运转"
```

**断点续跑**：所有状态持久化在数据库里，Gateway 恢复后编排层从数据库中的当前状态继续推进。正常的 tick 流程本身就是幂等的。

### 更鼓监控

| 指标 | 告警条件 | 处理 |
|---|---|---|
| 更鼓执行时间 | 单次 tick > 5 分钟 | 告警，检查是否有 Agent 卡住 |
| 更鼓跳过 | 连续 3 次 tick 未执行 | 告警，检查编排层进程 |
| 造物令 blocked | 进入 L4 暂停 | 告警，可能需要人工介入 |
| 降速中 | L3 降速超��� 2 小时 | 告警，可能不是偶发问题 |
| 封存率异常 | 连续 5 个造物令被封存 | 告警，检查 Agent prompt 或评分阈值 |

### 流程追溯

编排层的每一个决策和动作都记录在 events 表中，方便事后追溯"发生了什么、为什么、怎么处理的"。

#### 每次 tick 的执行摘要

每次更鼓触发时，编排层写入一条 `tick_executed` 事件：

```json
{
  "type": "tick_executed",
  "title": "更鼓第 48 声",
  "detail": {
    "tickNumber": 48,
    "projectId": "uuid",
    "projectName": "极简记账",
    "currentPhase": "builder",
    "action": "called_agent",
    "agentId": "jiangren",
    "taskId": "T-005",
    "result": "completed",
    "durationMs": 45000,
    "failCount": 0,
    "responseLevel": "normal"
  }
}
```

这样查询 `events WHERE type = 'tick_executed' AND project_id = ?` 就能看到一个造物令的完整推进历史——每次更鼓做了什么、调了谁、结果如何、耗时多少。

#### Agent 调用链路

每次调用 Agent 时，记录完整的调用链路：

```json
{
  "type": "agent_called",
  "detail": {
    "agentId": "jiangren",
    "sessionType": "isolated",
    "message": "完成 Hero 区的结构和布局",
    "projectPath": "~/.openclaw/products/jinji-jizhang/",
    "inputContext": { "phase": "builder", "task": "T-003", "section": "hero" },
    "startedAt": "2026-03-18T14:30:00Z"
  }
}
```

Agent 返回后，记录结果：

```json
{
  "type": "agent_called",
  "detail": {
    "agentId": "jiangren",
    "result": "completed",
    "outputSummary": "Hero 区结构完成，创建了 Hero.tsx 和 HeroImage.tsx",
    "filesChanged": ["src/components/sections/Hero.tsx", "src/components/sections/HeroImage.tsx"],
    "durationMs": 38000,
    "completedAt": "2026-03-18T14:30:38Z"
  }
}
```

#### 异常链路追踪

当出现异常时，从第一次失败到最终恢复/封存的完整链路都有记录：

```
events 时间线（按 created_at 排序）：

14:30  agent_called    调用匠人，任务 T-005
14:31  agent_failed    匠人超时，原因：Gateway 响应慢。fail_count: 1，级别: L1
14:31  agent_retried   L1 重试，同样指令再来一次
14:32  agent_called    重新调用匠人，任务 T-005
14:33  agent_failed    匠人再次超时。fail_count: 2，级别: L2
14:33  agent_retried   L2 调整重试，prompt 追加了上次错误信息
14:34  agent_called    调整后重新调用匠人
14:35  agent_failed    匠人第三次失败。fail_count: 3，级别: L3
14:35  backoff_triggered  降速：更鼓间隔从 20min 调整为 40min
15:15  agent_called    降速后重新调用匠人
15:16  agent_called    匠人成功完成 T-005
15:16  backoff_recovered  恢复正常间隔 20min，fail_count 归零
```

查询 `events WHERE project_id = ? AND type IN ('agent_failed', 'agent_retried', 'backoff_triggered', 'backoff_recovered') ORDER BY created_at` 就能看到完整的异常处理链路。

#### 阶段流转记录

每次阶段变更时记录决策依据：

```json
{
  "type": "phase_transition",
  "detail": {
    "from": "council",
    "to": "architect",
    "decision": "approved_with_conditions",
    "reason": "掌秤裁决：有条件通过",
    "conditions": ["MVP 3 天内完成", "核心差异化是零学习成本"],

  }
}
```

#### 追溯查询示例

| 想知道什么 | 怎么查 |
|---|---|
| 造物令 #7 的完整推进历史 | `events WHERE project_id = ? AND type = 'tick_executed' ORDER BY created_at` |
| 为什么造物令 #7 在锻造阶段卡了 2 小时 | `events WHERE project_id = ? AND type IN ('agent_failed', 'agent_retried', 'backoff_triggered') AND detail->>'currentPhase' = 'builder'` |
| 掌秤为什么放行了造物令 #7 | `events WHERE project_id = ? AND type = 'phase_transition' AND detail->>'from' = 'council'` |
| 匠人今天调用了多少次、成功率多少 | `events WHERE type = 'agent_called' AND detail->>'agentId' = 'jiangren' AND created_at > today()` |
| 最近有没有 Gateway 故障 | `events WHERE type IN ('gateway_down', 'gateway_recovered') ORDER BY created_at DESC` |
| 哪个阶段失败率最高 | `events WHERE type = 'agent_failed' GROUP BY detail->>'currentPhase'` |

#### 追溯的可见性

| 事件类型 | 展示网站可见？ | 用途 |
|---|---|---|
| tick_executed | ❌ 不可见 | 运维追溯 |
| agent_called | ❌ 不可见 | 运维追溯 |
| agent_failed | ❌ 不可见 | 运维追溯 |
| agent_retried | ⚠️ 转译后可见 | 群聊记录显示"局中人遇到了点小问题" |
| backoff_triggered | ⚠️ 转译后可见 | 群聊记录显示"造物流放慢了脚步" |
| project_blocked | ✅ 可见 | 造物坊看板显示"暂停中" |
| phase_transition | ✅ 可见 | 对话流中的阶段分隔线 |

运维级别的事件（tick_executed、agent_called、agent_failed）不在展示网站上显示——访客不需要看到"Gateway 返回 timeout"这种技术细节。但转译后的友好描述可以出现在群聊记录里。

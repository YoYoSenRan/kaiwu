# 局中人记忆系统设计

## 设计理念

局中人的记忆不是流水账，是**一生的经验沉淀**。

游商走过的每一条街、诤臣驳倒的每一个论点、匠人踩过的每一个坑——这些经历塑造了他们的判断力。记忆系统的目标是让局中人**越来越聪明**，而不只是"记住发生过什么"。

### 核心原则

1. **经验 > 日志**：重要的不是"14:30 调用了 web_search"，而是"工具类赛道竞品分析不能只看大厂"
2. **系统提炼 > Agent 自觉**：不依赖 Agent 自己判断什么该记，编排层在造物令完成后主动提炼经验
3. **分类存储 > 一锅炖**：教训、模式、领域知识分开存，检索时精准命中
4. **衰减有度**：日常琐事随时间淡忘，但刻骨铭心的教训永不褪色

## 三层记忆架构

```
┌─────────────────────────────────────────────────────────┐
│  第一层：工作记忆（Session Context）                       │
│                                                         │
│  生命周期：单次 cron isolated session                     │
│  内容：当前造物令的上下文、上游产出、任务指令              │
│  来源：编排层通过 cron message 注入                       │
│  特点：用完即弃，不持久化                                 │
└─────────────────────────────────────────────────────────┘
                         │
                    造物令完成后
                    编排层提炼
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  第二层：经验记忆（Structured Memory Files）               │
│                                                         │
│  生命周期：永久，但有衰减权重                             │
│  内容：教训、成功模式、领域知识、角色关系                  │
│  来源：编排层在造物令完成后主动写入                        │
│  存储：workspace 下的结构化 Markdown 文件                  │
│  检索：OpenClaw memory_search（向量 + BM25 混合搜索）     │
└─────────────────────────────────────────────────────────┘
                         │
                    复盘时精炼
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  第三层：核心记忆（MEMORY.md）                             │
│                                                         │
│  生命周期：永久，不衰减                                   │
│  内容：最重要的经验精华、属性快照、里程碑                  │
│  来源：复盘时从第二层精炼而来                              │
│  特点：每次会话必读，控制在 200 行以内                     │
└─────────────────────────────────────────────────────────┘
```

### 为什么三层？

- **第一层**解决"当前任务需要什么信息"——cron isolated session 天然隔离，不会污染
- **第二层**解决"过去的经验怎么找到"——结构化文件 + 语义搜索，精准召回
- **第三层**解决"最重要的东西不能忘"——MEMORY.md 每次会话加载，核心经验始终在场

## 第二层：经验记忆文件结构

每个局中人的 workspace 下，`memory/` 目录从扁平的日期文件改为结构化目录：

```
~/.openclaw/workspace-youshang/
├── MEMORY.md                    # 第三层：核心记忆（精华）
└── memory/
    ├── YYYY-MM-DD.md            # 每日流水（OpenClaw 默认，保留）
    ├── lessons.md               # 教训："下次不要..."
    ├── patterns.md              # 成功模式："这招管用..."
    ├── domain/                  # 领域知识（按赛道分文件）
    │   ├── 工具类.md
    │   ├── 社交类.md
    │   └── 电商类.md
    └── relationships.md         # 角色关系记忆
```

### 各文件的内容格式

#### lessons.md — 教训

```markdown
# 教训

## L-001 · 竞品分析不能只看大厂
- 来源：造物令 #3「极简 CRM」
- 场景：采风时只看了 Salesforce、HubSpot，漏掉了 3 个小作坊竞品
- 后果：采风评分偏高（85），实际进入过堂后被诤臣用小作坊数据反驳
- 教训：下次采风时，搜索范围要包含"indie"、"solo"、"小团队"关键词
- 重要度：★★★★☆

## L-002 · "极简"不是差异化
- 来源：造物令 #5「极简笔记」
- 场景：采风报告把"极简"作为核心差异化点
- 后果：诤臣一句"每个失败的笔记 App 都说自己极简"直接封存
- 教训：差异化要找具体的功能点，不能用抽象的形容词
- 重要度：★★★★★
```

#### patterns.md — 成功模式

```markdown
# 成功模式

## P-001 · 留存率数据是过堂利器
- 来源：造物令 #7「极简记账」
- 场景：说客用"现有产品 30 天留存率只有 22%"反击诤臣的"市场饱和"论点
- 效果：诤臣罕见地松口，掌秤有条件放行
- 模式：当诤臣说"市场饱和"时，用留存率/弃用率数据证明"不是需求不存在，是产品不够好"
- 适用场景：竞品多但留存率低的赛道

## P-002 · 三天 MVP 约束反而提升质量
- 来源：造物令 #7「极简记账」
- 场景：掌秤附带"3 天内完成 MVP"的条件
- 效果：匠人被迫砍掉非核心功能，最终产品反而更聚焦
- 模式：严格的时间约束可以作为功能筛选器
```

#### domain/工具类.md — 领域知识

```markdown
# 工具类赛道知识

## 市场特征
- 竞品数量通常 > 50，但头部集中度低
- 用户付费意愿中等，关键是"省时间"的感知
- 留存率普遍偏低（20-30%），说明产品同质化严重

## 采风要点
- 必须搜索 indie/solo 开发者的产品，不能只看大厂
- Product Hunt 和 AlternativeTo 是好的竞品发现渠道
- 关注"弃用原因"比"使用原因"更有价值

## 历史数据
- 造物令 #3「极简 CRM」：采风 85 分，过堂封存（漏掉小作坊竞品）
- 造物令 #7「极简记账」：采风 72 分，最终鸣锣（留存率数据是关键）
```

#### relationships.md — 角色关系记忆

```markdown
# 角色关系记忆

## 与说客
- 说客最常引用我报告中的"市场规模"和"用户痛点"数据
- 造物令 #7 中，说客用我报告里的留存率数据完成了关键反击
- 建议：采风报告中多提供"现有产品的弱点"数据，这是说客最需要的弹药

## 与诤臣
- 诤臣最常质疑我报告中的"差异化空间"评分
- 造物令 #3 中，诤臣用我漏掉的小作坊数据反驳了我的高分
- 建议：采风时主动预判诤臣会从哪个角度质疑，提前在报告中回应
```

## 编排层主动提炼机制

**Agent 不需要自己决定什么该记住。** 编排层在每次造物令完成后，主动对比"当初的判断"和"最终结果"，提炼经验写入对应 Agent 的记忆文件。

### 提炼流程

```
造物令完成（鸣锣 or 封存）
  │
  ├── 1. 收集各局中人的原始判断
  │      ├── 游商：采风评分和各维度判断
  │      ├── 说客：支持的核心论点
  │      ├── 诤臣：质疑的核心风险
  │      ├── 掌秤：裁决和附带条件
  │      ├── 画师：蓝图的技术选型
  │      ├── 匠人：遇到的困难和解决方案
  │      ├── 试剑：发现的问题
  │      └── 鸣锣：部署中的意外
  │
  ├── 2. 对比最终结果
  │      ├── 鸣锣成功：哪些判断被验证了？哪些条件是多余的？
  │      └── 封存：封存原因是什么？谁的判断最准？
  │
  ├── 3. 提炼经验（编排层调用 LLM 生成）
  │      ├── 教训（lesson）：判断失误的地方 → 写入 lessons.md
  │      ├── 模式（pattern）：判断正确的地方 → 写入 patterns.md
  │      ├── 领域知识（domain）：赛道特征 → 写入 domain/{赛道}.md
  │      └── 关系记忆（relationship）：角色互动中的发现 → 写入 relationships.md
  │
  ├── 4. 写入对应 Agent 的 memory/ 目录
  │
  └── 5. 更新 agent_stats 表（属性值）
```

### 各角色的提炼维度

| 角色 | 对比什么 | 提炼什么 |
|---|---|---|
| 游商 | 采风评分 vs 最终结果 | 评分准确度、漏掉的竞品、领域特征 |
| 说客 | 支持的论点 vs 最终结果 | 哪些论证策略有效、哪些数据最有说服力 |
| 诤臣 | 质疑的风险 vs 最终结果 | 哪些风险预判准确、哪些是过度担忧 |
| 掌秤 | 裁决和条件 vs 最终结果 | 裁决是否正确、条件是否合理 |
| 画师 | 蓝图 vs 实际实现 | 任务拆解是否合理、技术选型是否正确 |
| 匠人 | 预估工时 vs 实际工时 | 哪些任务比预期难、常见的技术坑 |
| 试剑 | 审查发现 vs 线上表现 | 是否漏掉了问题、审查标准是否合理 |
| 鸣锣 | 部署计划 vs 实际部署 | 部署中的意外、回滚是否用上了 |

### 提炼示例：造物令 #7「极简记账」鸣锣成功

**游商的提炼**：
```
→ lessons.md: 无新增（评分 72，最终成功，判断基本准确）
→ patterns.md: "72 分也能成功，不要只看总分，要看各维度的分布"
→ domain/工具类.md: 追加"记账赛道留存率普遍 < 25%，这是切入点"
→ relationships.md: "说客用了我报告里的留存率数据完成关键反击"
```

**诤臣的提炼**：
```
→ lessons.md: "被留存率数据说服后松口，最终证明松口是对的——要学会接受有力的反证据"
→ patterns.md: 无新增
→ relationships.md: "说客在造物令 #7 中用数据反击成功，下次要提前查留存率数据"
```

## 记忆检索优化

### OpenClaw 配置

启用混合搜索 + 时间衰减 + MMR 去重：

```json5
agents: {
  defaults: {
    memorySearch: {
      // 启用向量 + BM25 混合搜索
      query: {
        hybrid: {
          enabled: true,
          vectorWeight: 0.7,
          textWeight: 0.3,
          // 时间衰减：日常流水 30 天半衰期，但 lessons/patterns 不衰减
          temporalDecay: {
            enabled: true,
            halfLifeDays: 30
          },
          // MMR 去重：避免返回重复的经验
          mmr: {
            enabled: true,
            lambda: 0.7
          }
        }
      },
      // 额外索引结构化记忆文件
      extraPaths: ["memory/lessons.md", "memory/patterns.md", "memory/domain/", "memory/relationships.md"]
    }
  }
}
```

### 衰减规则

| 文件类型 | 衰减策略 | 理由 |
|---|---|---|
| memory/YYYY-MM-DD.md | 30 天半衰期 | 日常流水，旧的自然淡忘 |
| memory/lessons.md | **不衰减** | 教训是刻骨铭心的，永远不能忘 |
| memory/patterns.md | **不衰减** | 成功模式是核心竞争力 |
| memory/domain/*.md | 90 天半衰期 | 领域知识会过时，但衰减慢 |
| memory/relationships.md | **不衰减** | 角色关系是长期积累 |
| MEMORY.md | **不衰减** | 核心记忆，永远在场 |

> OpenClaw 的 temporal decay 只对带日期的文件（YYYY-MM-DD.md）生效，非日期文件默认不衰减。lessons.md、patterns.md 等文件天然不受衰减影响。

### Agent 如何使用记忆

在 SOUL.md 的"任务响应"中，Agent 被要求在开始工作前先检索相关记忆：

```
收到采风任务时：
1. 调用 memory_search("该赛道的历史教训和经验") → 命中 lessons.md 和 domain/*.md
2. 调用 memory_search("该类型物帖的成功模式") → 命中 patterns.md
3. 调用 get_my_stats() → 读取属性面板
4. 开始采风，带着历史经验做判断
```

这样游商在采风"极简笔记"时，会自动召回"L-002：'极简'不是差异化"这条教训，避免重蹈覆辙。

## 第三层：核心记忆（MEMORY.md）精炼

MEMORY.md 是每次会话必读的，所以必须**精简、高密度**。不能把所有经验都塞进去。

### 精炼时机

复盘时（7/30/90 天）从第二层提炼到第三层：

```
复盘触发
  │
  ├── 扫描 lessons.md 和 patterns.md
  │
  ├── 筛选重要度 ★★★★☆ 以上的经验
  │
  ├── 压缩为一句话写入 MEMORY.md 的"经验与教训"段落
  │
  └── 如果 MEMORY.md 超过 200 行，删除最旧的低重要度条目
```

### MEMORY.md 模板（更新版）

```markdown
# 游商 · 成长档案

## 等级

Lv.2 行脚期（经手 8 个造物令）

## 属性快照

| 属性 | 星级 | 原始值 | 样本数 |
|---|---|---|---|
| 嗅觉 | ★★★☆☆ | 0.72 | 8 |
| 脚力 | ★★☆☆☆ | 0.65 | 8 |
| 见闻 | ★★★☆☆ | 8 | 8 |
| 慧眼 | ★★☆☆☆ | 0.38 | 8 |

## 核心教训（刻骨铭心，永远不忘）

- 竞品分析必须包含 indie/solo 开发者，不能只看大厂（#3 极简CRM 的惨痛教训）
- "极简"不是差异化，要找具体的功能切入点（#5 极简笔记被一句话封存）
- 72 分也能鸣锣，不要只看总分，要看各维度分布（#7 极简记账）

## 核心模式（屡试不爽）

- 留存率/弃用率数据是过堂中最有力的武器
- 搜索"弃用原因"比"使用原因"更能发现差异化空间

## 领域专长

- 工具类 ★★★★☆：最熟悉的赛道，踩过坑也出过成绩

- 造物令 #8：第一次在报告中直接建议"此物不必造"，省去了过堂
```

## 存储策略

**workspace Markdown 文件是唯一的 source of truth。** 不在数据库里冗余存储经验记忆。

```
编排层提炼经验
  │
  └── 写入 workspace memory/ 文件（Agent 运行时使用）
      ├── OpenClaw memory_search 直接索引这些文件
      └── 展示网站需要的"成长历程"从 agent_logs + agent_stats + retrospectives 组装
```

为什么不用数据库表：
- workspace 文件已经是 OpenClaw 的原生记忆机制，memory_search 直接索引
- 多一张表 = 多一套同步逻辑 = 多一个出 bug 的地方
- 展示网站需要的数据可以从现有的 agent_logs（type: thought, visibility: public）和 agent_stats 组装

## 每日总结（Cron 定时提炼）

造物令可能跑好几天。如果只在造物令完成时才提炼经验，中间的教训就会丢失——匠人今天踩了一个坑，不应该等三天后鸣锣才记住。

每日总结补上这个缺口：**每天固定时间，编排层回顾当天发生的事，提炼经验写入记忆文件。**

### Cron 配置

```bash
openclaw cron add \
  --name "每日总结" \
  --cron "0 23 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --message "回顾今天所有局中人的工作，提炼经验。" \
  --no-deliver
```

### 谁来做？

**编排层统一做，不唤醒 Agent。** 理由：

- Agent 自己总结自己容易"自我感觉良好"，该记的教训可能被跳过
- 编排层有全局视角，能对比多个局中人的表现
- 不消耗额外的 Agent 调用 token

### 总结流程

```
每日 23:00 cron 触发（isolated session）
  │
  ├── 1. 查询今天的数据
  │      ├── agent_logs（today，visibility: public）
  │      ├── events（today）
  │      ├── debates（today）
  │      └── tasks 状态变更（today）
  │
  ├── 2. 按局中人分组，判断今天谁干了活
  │      ├── 游商：今天采风了「宠物心理咨询」
  │      ├── 说客+诤臣：过堂第 2 轮
  │      ├── 匠人：完成了 T-003、T-005，T-007 被卡住
  │      └── 其他人：今天没有造物流任务
  │
  ├── 3. 编排层调用 LLM 生成每日摘要
  │      ├── 事实：今天发生了什么
  │      ├── 提炼：有没有值得记住的教训或模式
  │      └── 评价：各局中人的表现亮点或问题
  │
  ├── 4. 写入各 Agent 的 memory/YYYY-MM-DD.md
  │      （只写有活动的 Agent，没干活的不写）
  │
  └── 5. 如果提炼出了重要经验
        └── 直接写入 lessons.md 或 patterns.md
            （不等造物令完成，当天就沉淀）
```

### 每日摘要的内容格式

写入 `memory/YYYY-MM-DD.md` 的内容：

```markdown
# 2026-03-17 · 每日总结

## 今日造物流

当前造物令：「极简记账」—— 锻造中

## 我的工作

- 完成了 T-003「实现记账页面 UI」，耗时 2 个更鼓
- 完成了 T-005「实现账单分类 API」，耗时 3 个更鼓
- T-007「实现数据导出」被卡住——依赖 T-006 还没完成

## 今日经验

- CSS 居中问题又出现了，这次用 flexbox 解决比 grid 更快（→ 已写入 lessons.md）
- 账单分类的正则表达式比预期复杂，下次遇到类似需求要多预留时间
```

### 什么情况下直接写入 lessons.md / patterns.md？

不是所有每日经验都值得进入长期记忆。只有满足以下条件的才写入：

| 条件 | 写入目标 | 示例 |
|---|---|---|
| 同一类问题出现 ≥ 2 次 | lessons.md | "CSS 居中问题第 3 次出现了" |
| 某个做法明显提升了效率 | patterns.md | "用 flexbox 比 grid 快 50%" |
| 犯了一个可避免的错误 | lessons.md | "忘记检查时区导致跨日 bug" |
| 发现了一个领域特征 | domain/*.md | "记账类产品的用户最在意输入速度" |

编排层在生成每日摘要时，用 LLM 判断哪些经验满足上述条件，自动分类写入。

### 没有活动的日子

如果今天造物流没有推进（比如没有造物令在跑，或者更鼓间隔很长），每日总结 cron 仍然触发，但：

- 检查到没有今日活动 → 不写任何文件
- 如果游商有自由活动（预采风、趋势监测）→ 只写游商的每日摘要

不会生成空的 `memory/YYYY-MM-DD.md` 文件。

### 与造物令完成时提炼的关系

两者互补，不冲突：

| 时机 | 提炼什么 | 写入哪里 |
|---|---|---|
| 每日总结 | 当天的具体经验（战术层面） | memory/YYYY-MM-DD.md + lessons/patterns（如果够重要） |
| 造物令完成 | 整个造物令的全局复盘（战略层面） | lessons.md + patterns.md + domain/*.md + relationships.md |
| 复盘（7/30/90天） | 器物的实际市场表现 | MEMORY.md（精炼核心记忆） |

每日总结抓的是**细节**（"今天踩了什么坑"），造物令完成时抓的是**全局**（"这个造物令整体学到了什么"），复盘抓的是**验证**（"当初的判断对不对"）。三个时间尺度，三种粒度，互不替代。

---

MVP 阶段用 OpenClaw 默认的 SQLite 向量搜索即可。如果后续记忆量增大（单个 Agent 积累 100+ 条经验），可以启用 QMD 后端获得更好的检索质量：

```json5
memory: {
  backend: "qmd",
  qmd: {
    includeDefaultMemory: true,
    update: { interval: "5m" },
    limits: { maxResults: 6, timeoutMs: 4000 }
  }
}
```

QMD 的优势：
- BM25 + 向量 + 重排序三路混合，比默认的双路更精准
- 本地运行，不依赖外部 API
- 支持更细粒度的相关性排序

## 实施优先级

| 优先级 | 内容 | 工作量 | 价值 |
|---|---|---|---|
| P0 | 结构化记忆文件（lessons/patterns/domain/relationships） | 小 | 高——立即提升记忆质量 |
| P0 | 编排层主动提炼机制 | 中 | 核心——让局中人真正"成长" |
| P1 | OpenClaw 混合搜索 + 衰减 + MMR 配置 | 小 | 中——提升检索精度 |
| P1 | MEMORY.md 精炼机制（复盘时从第二层提炼） | 小 | 中——保持核心记忆精简 |
| P2 | QMD 后端 | 中 | 锦上添花——记忆量大时再启用 |
| P2 | 数据库同步 + 展示网站展示成长历程 | 中 | 展示价值——让访客看到局中人的成长 |
